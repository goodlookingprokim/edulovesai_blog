## 요구사항

### STEP 1: 정보 수집 (Information Gathering)
1. 레퍼런스 파일 읽기
   - '/Users/ohkyunga/Downloads/공유받은자료/허민님 자료/허민님옵시프롬프트2507.md' 전체 읽기
   - 옵시디언 노트 구조, PARA 시스템, 메타데이터 형식 파악

2. 소스 파일 분석
   - [STT 파일 경로] 디렉토리의 모든 파일 나열
   - 각 파일의 전체 내용 읽기 (긴 파일은 여러 번에 나누어 읽기)
   - 총 라인 수 확인

### STEP 2: 정보 추출 (Information Extraction)
**중요**: 정보 손실 없이 100% 모든 내용을 추출해야 합니다.

다음 항목을 빠짐없이 추출:
- 모든 대화 내용과 발언자
- 숫자, 금액, 날짜 등 구체적 수치
- 인명, 회사명, 도구명, 기술명
- 직접 인용문 (원문 그대로)
- 예시 코드, 명령어, 설정 방법
- 워크플로우, 프로세스, 단계별 설명
- 비교 대상 (vs, 대비 등)
- URL, 링크, 참고 자료

### STEP 3: 구조화 (Structuring)
1. 논리적 섹션 분류
   - 주요 주제별로 대분류 (## 레벨)
   - 세부 주제별로 중분류 (### 레벨)
   - 구체적 내용별로 소분류 (#### 레벨)

2. 내용 조직화
   - 시간 순서가 있으면 시간순 배열
   - 개념 설명 → 실전 활용 → 주의사항 순서
   - 관련 내용끼리 그룹화

### STEP 4: 시각화 추가 (Visualization)
**중요**: flow나 비교 등이 필요한 경우 시각화를 포함해서 문서 작성해

다음 경우 Mermaid 다이어그램 또는 비교표 추가:

#### 1. 워크플로우 (Flowchart/Graph)
- 순차적 프로세스가 있는 경우
- 의사결정 트리가 있는 경우
- 시스템 아키텍처 설명

예시:
\```mermaid
graph TD
    A[시작] --> B{조건}
    B -->|Yes| C[작업 A]
    B -->|No| D[작업 B]
    C --> E[완료]
    D --> E
\```

#### 2. 상태 전이 (State Diagram)
- 상태 변화가 있는 경우
- 모드 전환 설명

예시:
\```mermaid
stateDiagram-v2
    [*] --> Idle
    Idle --> Running: 시작
    Running --> Complete: 성공
    Running --> Error: 실패
\```

#### 3. 비교표 (Comparison Table)
- 2개 이상의 옵션/도구/방법 비교
- 성능, 비용, 기능 차이 설명

예시:
| 구분 | 옵션 A | 옵션 B | 차이점 |
|------|--------|--------|--------|
| 비용 | 10만원 | 1만원 | 90% 절감 |
| 속도 | 빠름 | 중간 | - |
| 난이도 | 어려움 | 쉬움 | - |

#### 4. 아키텍처 (Architecture Diagram)
- 시스템 구성 설명
- 모듈 간 관계
- 데이터 흐름

예시:
\```mermaid
graph LR
    subgraph Frontend
        A[UI]
    end
    subgraph Backend
        B[API]
        C[DB]
    end
    A --> B
    B --> C
\```

## 작업 원칙

### 정보 보존 원칙
✅ **반드시 지켜야 할 것**:
- 모든 숫자, 금액, 날짜 원본 그대로 보존
- 인명, 회사명, 도구명 정확히 기록
- 직접 인용문은 인용 블록(>)으로 표시
- 예시 코드는 코드 블록(\```)으로 표시
- 구체적 사례, 예시는 전부 포함

❌ **하지 말아야 할 것**:
- 내용 요약하여 생략
- 중요하지 않다고 판단하여 삭제
- 비슷한 내용 통합하여 정보 손실
- 추측이나 해석 추가
- 원문에 없는 내용 생성

### 구조화 원칙
✅ **좋은 구조**:
- 논리적 흐름 (개념 → 활용 → 주의사항)
- 명확한 계층 (##, ###, ####)
- 시각화로 복잡한 내용 단순화
- 비교표로 차이점 명확화

❌ **나쁜 구조**:
- 시간 순서만 따라가기
- 관련 없는 내용 혼재
- 시각화 누락
- 단순 나열식 구성

### 시각화 원칙
✅ **시각화 추가 기준**:
- 3단계 이상의 프로세스
- 2개 이상의 비교 대상
- 상태 전이가 있는 경우
- 시스템 아키텍처 설명
- 의사결정 트리

❌ **시각화 불필요**:
- 단순 나열 정보
- 1-2줄로 설명 가능한 내용
- 이미 표로 정리된 내용

## 트러블슈팅

### 문제: 파일이 너무 길어서 한 번에 못 읽음
**해결**: 파일을 여러 번에 나누어 읽기 (offset, limit 사용)
\```
Read 파일 (offset=0, limit=200)
Read 파일 (offset=200, limit=200)
...
\```

### 문제: 시각화를 언제 추가해야 할지 모르겠음
**해결**: 다음 키워드가 나오면 무조건 추가
- "워크플로우", "프로세스", "단계", "흐름"
- "vs", "대비", "비교", "차이"
- "상태", "전이", "모드"
- "아키텍처", "구조", "시스템"
- "비용", "성능", "속도" (수치 비교)

### 문제: 정보가 너무 많아서 정리가 안 됨
**해결**:
1. 일단 모든 내용을 섹션별로 분류
2. 각 섹션 내에서 소주제로 재분류
3. 시각화로 복잡한 부분 단순화
4. 비교표로 옵션 정리
