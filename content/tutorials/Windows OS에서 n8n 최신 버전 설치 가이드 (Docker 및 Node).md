# 윈도우 환경에서 n8n 로컬 설치 가이드 (Docker 및 Node.js 방법)

**개요:** n8n은 다양한 서비스를 손쉽게 연결하여 워크플로우 자동화를 구현할 수 있는 오픈소스 툴입니다. Windows PC에 n8n을 로컬로 설치하는 방법에는 **Docker 기반 설치**와 **Node.js (npm) 직접 설치** 두 가지가 있습니다. Docker 방법은 격리된 환경과 쉬운 업그레이드 등으로 **권장되는 방식**이며, Node.js 직접 설치는 Docker 사용이 어려운 경우에 **대안**이 됩니다. 아래에서는 각 방법에 대하여 **준비 사항**, **설치 단계**, **환경 설정 (.env, 포트, 사용자 정보 등)**, **실행 및 테스트 방법**, 그리고 **장단점**을 초보자도 따라할 수 있게 단계별로 설명합니다.

## 1\. Docker를 이용한 n8n 설치 (WSL2 기반)

**Docker 방법 개요:** Docker 컨테이너를 이용하면 n8n과 그 **종속성**을 격리된 환경에서 실행할 수 있어 충돌을 방지하고, 다른 시스템으로의 이식도 쉬워집니다. 또한 백그라운드 서비스로 실행하거나 이미지 업데이트로 손쉽게 업그레이드할 수 있다는 장점이 있습니다. Windows에서 Docker로 n8n을 실행할 때에는 \*\*WSL 2 (Windows Subsystem for Linux)\*\*를 활용하며, 이를 통해 Linux 기반 컨테이너를 원활히 구동합니다. 아래 절차에 따라 설치를 진행하세요.

### 1.1 설치 전 준비 사항

* **운영체제:** Windows 10 또는 Windows 11 (64비트). 최신 윈도우 업데이트 적용을 권장합니다.  
* **관리자 권한:** 소프트웨어 설치 및 설정 변경을 위해 PC의 관리자 권한이 필요합니다.  
* **가상화 활성화:** **WSL2** 기반 Docker를 사용하려면 PC의 BIOS/UEFI에서 하드웨어 가상화(예: Intel VT-x 또는 AMD-V)를 **Enabled(활성화)** 해야 합니다. 또한 Windows 기능에서 "가상 머신 플랫폼"과 "Linux용 Windows 하위 시스템"을 켜야 합니다. *(TIP: BIOS 설정 변경에 익숙하지 않을 경우 Node.js 직접 설치 방법을 고려하세요.)*  
* **Docker Desktop:** Docker 컨테이너 실행을 위해 **Docker Desktop** 프로그램을 설치해야 합니다. Docker Hub 계정이 필요하며(무료 가입 가능), 설치 과정에서 WSL 2 옵션을 선택합니다.

### 1.2 Docker Desktop 설치 및 WSL2 설정

1. **Docker Desktop 다운로드 및 설치:** Docker 공식 웹사이트에 로그인한 후 Windows용 Docker Desktop 설치 프로그램을 내려받습니다. 설치 마법사의 안내에 따라 Docker Desktop을 설치하세요. 도중에 "**Enable WSL 2 instead of Hyper-V**" 옵션이 나타나면 체크합니다 (기본값으로 활성화됨).  
     
2. **시스템 재시작:** Docker Desktop 설치 후에는 PC를 한 번 **재부팅**합니다. (※ 재부팅을 통해 WSL2 커널 및 Docker 서비스가 정상적으로 적용됩니다.)  
     
3. **WSL2 기능 확인:** 재시작 후 Docker Desktop을 실행합니다. 정상적으로 시작되지 않거나 WSL 관련 오류가 나타난다면, 다음을 확인하세요:  
     
   * **WSL 업데이트:** 관리자 권한 PowerShell에서 `wsl --update` 명령으로 WSL 구성 요소를 업데이트합니다. 완료 후 `wsl --shutdown`으로 WSL을 재시작합니다.  
   * **Windows 기능 확인:** *Windows 기능 켜기/끄기* 메뉴에서 "Virtual Machine Platform(가상 머신 플랫폼)"과 "Windows Subsystem for Linux(WSL)" 항목이 활성화되어 있는지 확인하고, 꺼져 있다면 활성화 후 재부팅합니다.  
   * **BIOS 가상화 설정:** WSL2 오류 메시지에 “**virtualization is not enabled in the BIOS**” 또는 `HCS_E_HYPERV_NOT_INSTALLED`와 같은 문구가 포함되면, 이는 BIOS에서 가상화 기술이 꺼져 있음을 의미합니다. 이 경우 컴퓨터를 재부팅하여 BIOS 설정 화면에 들어가 **Intel VT-x**, **AMD-V**, **SVM Mode** 등의 옵션을 Enabled로 변경한 뒤 저장하고 나오세요. (제조사별로 BIOS 진입 키가 다르며, 일반적으로 F2, F10, Del 등을 사용합니다.)

**참고:** Docker Desktop 설치가 완료되고 WSL2가 정상 동작하면, 작업표시줄의 Docker 아이콘이 **Running** 상태로 표시됩니다. 이 상태에서 다음 단계로 진행합니다.

### 1.3 n8n 컨테이너 실행 및 환경 설정

이제 Docker로 n8n을 실행하겠습니다. 실행 시 **데이터 영속성**을 위해 Windows의 특정 폴더를 컨테이너 내부의 n8n 데이터 디렉토리에 연결할 것입니다. 기본적으로 n8n은 사용자 홈 디렉터리에 `.n8n` 폴더를 만들어 설정, 워크플로우, 자격증명 등을 보관합니다. Docker 사용 시 이 폴더를 호스트와 공유하거나 Docker 볼륨을 사용해야 컨테이너 재시작 후에도 데이터가 유지됩니다.

* **데이터 폴더 준비:** 특별한 작업은 필요 없으며, Docker 실행 시 경로를 지정하면 없을 경우 자동 생성됩니다. 기본 경로는 `C:\Users\<YourUser>\.n8n` (사용자 홈의 `.n8n` 폴더)입니다. 이 경로에 n8n의 SQLite DB, 암호화키 등 중요한 데이터가 저장되므로 지우지 않도록 주의하세요.

**Docker에서 n8n 실행:** PowerShell을 **관리자 권한**으로 열고, 아래 명령어를 입력하여 n8n 컨테이너를 실행합니다 (첫 실행 시 필요한 이미지를 자동으로 다운로드합니다).

docker run \-d \--name n8n \-p 5678:5678 \-e N8N\_RUNNERS\_ENABLED=true \-v ${env:USERPROFILE}\\.n8n:/home/node/.n8n docker.n8n.io/n8nio/n8n:latest

위 명령의 각 옵션 의미는 다음과 같습니다:

* `-d`: \*\*데몬 모드(백그라운드 실행)\*\*로 컨테이너를 실행합니다. 터미널을 닫아도 n8n 프로세스는 백그라운드에서 계속 동작합니다.  
* `--name n8n`: 컨테이너 이름을 "n8n"으로 지정합니다. 이후 컨테이너 관리 시 이름으로 참조할 수 있어 편리합니다.  
* `-p 5678:5678`: 호스트의 포트 5678을 컨테이너의 5678포트에 바인딩합니다. n8n의 기본 웹 인터페이스 포트는 **5678**이며, 필요시 변경 가능합니다 (뒷부분 참고).  
* `-e N8N_RUNNERS_ENABLED=true`: n8n의 워크플로우 멀티스레드 실행 기능을 활성화합니다. *(선택 사항이지만 최신 버전에서 성능 향상을 위해 권장)*  
* `-v ${env:USERPROFILE}\.n8n:/home/node/.n8n`: **데이터 볼륨 마운트** 옵션으로, Windows 사용자 프로필 폴더의 `.n8n` 디렉터리를 컨테이너 내부의 `/home/node/.n8n` 경로에 연결합니다. 이를 통해 **워크플로우, 설정, 암호화 키 등의 데이터가 호스트에 영구 저장**되어 컨테이너를 재시작/재생성해도 데이터가 유지됩니다.  
* `docker.n8n.io/n8nio/n8n:latest`: n8n 공식 Docker 이미지의 **latest 태그**를 사용합니다. (최신 안정 버전이 실행됨)

명령어 입력 후 Enter를 누르면 Docker가 이미지를 다운로드하고 컨테이너를 생성합니다. 완료되면 n8n 서버가 컨테이너 내부에서 백그라운드로 구동됩니다. Docker Desktop의 대시보드에서 컨테이너 상태를 확인할 수도 있지만, 별도로 Docker Desktop에서 Run을 누를 필요는 없습니다.

* **포트 및 네트워크:** 위에서 호스트 포트와 컨테이너 포트를 동일하게 5678로 매핑했으므로, 본인 PC에서만 접근할 경우 그대로 사용하면 됩니다. 만약 **다른 포트로 변경**하고 싶다면 `-p <호스트포트>:5678` 형식으로 앞의 숫자를 변경하거나, 환경 변수 `N8N_PORT`를 컨테이너에 설정할 수 있습니다. 예를 들어 8080 포트를 사용하고 싶다면 `-p 8080:5678`로 실행한 뒤 브라우저에서 localhost:8080으로 접속하면 됩니다.  
    
* **환경 변수 파일(.env) 사용:** 여러 환경 변수를 한꺼번에 설정해야 하는 경우, `.env` 파일을 만들어 변수들을 작성해두고 `--env-file` 옵션으로 적용할 수 있습니다. 또는 Docker Compose를 이용하여 `docker-compose.yml`과 `.env` 파일에 환경 변수를 관리할 수 있습니다. (예: `N8N_PORT=8080`, `N8N_PROTOCOL=https` 등의 키=값을 `.env`에 지정). 초보자의 경우 기본 설정으로도 무방하므로, 필요한 경우에만 활용합니다.  
    
* **초기 사용자 계정:** 컨테이너의 n8n을 처음 실행하면 내부적으로 SQLite 데이터베이스와 암호화 키가 생성되고, 웹 UI에 접속했을 때 **소유자(관리자) 계정 생성**을 요청합니다. 초반 실행 시 로그에 초기화 관련 메시지가 나타날 수 있으며, 최초 접속 시 입력한 이메일/비밀번호가 관리자 계정으로 설정되어 `.n8n` 폴더 내 DB에 저장됩니다.

### 1.4 실행 확인 및 웹 인터페이스 접속

컨테이너가 시작되고 약간의 초기화 시간을 가진 후, 웹 브라우저를 열어 [**http://localhost:5678**](http://localhost:5678) 주소로 접속합니다. (만약 다른 포트를 지정했다면 해당 포트로 접속) n8n 로컬 인스턴스의 **초기 설정 화면**이 나타나면, 안내에 따라 관리자 이메일, 이름, 비밀번호 등을 입력하여 계정을 생성합니다. **"업데이트 알림 수신"** 등의 개인화 질문이 나올 수 있는데, 필수는 아니므로 원하는 대로 응답합니다. 계정 생성 후 **n8n 에디터 UI**가 표시되며, 이제 로컬에서 n8n을 자유롭게 사용할 수 있습니다.

설정한 계정 정보와 생성한 워크플로우 등은 모두 앞서 마운트한 `.n8n` 폴더에 저장되므로, 컨테이너를 정지하거나 삭제해도 데이터는 유지됩니다. 예를 들어 `C:\Users\<User>\.n8n` 경로에 데이터베이스 파일(`database.sqlite` 등)과 암호화 키(`encryptionKey`) 파일이 생성되어 있습니다. **보안 주의:** 이 폴더를 백업하면 워크플로우와 자격증명을 복원할 수 있지만, 다른 사람에게 유출되지 않도록 하세요 (암호화 키 유실 시 기존 자격증명 복호화 불가).

### 1.5 n8n 컨테이너 관리 및 종료

Docker로 실행한 n8n은 백그라운드에서 계속 동작하기 때문에, PC를 켜는 동안 자동 실행되길 원하면 Docker Desktop 설정에서 **Windows 시작 시 Docker 자동 실행** 옵션을 켜둘 수 있습니다. 수동으로 컨테이너를 관리할 경우 다음 명령어들을 PowerShell에서 사용할 수 있습니다:

* `docker ps` : 실행 중인 컨테이너 목록을 확인 (n8n 컨테이너의 ID와 이름을 확인 가능).  
* `docker stop n8n` : 이름이 "n8n"인 컨테이너를 **중지**합니다. (일시적으로 멈출 때)  
* `docker start n8n` : 중지된 n8n 컨테이너를 **재시작**합니다. (데이터는 그대로 유지됨)  
* `docker logs n8n` : 컨테이너 로그 출력. 추가로 `-f` 옵션을 붙이면 실시간 로그를 팔로우합니다.  
* `docker restart n8n` : 컨테이너를 재시작 (중지 후 바로 시작)을 한 번에 수행합니다.  
* `docker rm n8n` : 컨테이너를 삭제합니다. (삭제해도 `\.n8n` 볼륨 데이터는 남지만, 동일 이름으로 새 컨테이너 생성 시 충돌 방지를 위해 필요할 때 사용) *(참고: 삭제 전에 `docker stop n8n`으로 중지 필요. `-f` 옵션으로 강제 삭제도 가능)*

이외에도 Docker Desktop GUI에서 **Containers / Apps** 메뉴에서 n8n 컨테이너를 선택해 Start/Stop할 수 있고, **Images** 메뉴에서 최신 이미지를 Pull하여 업데이트할 수도 있습니다. 업데이트 시에는 새로운 이미지를 받은 후 기존 컨테이너를 삭제(`docker rm`)하고 위 `docker run` 명령을 다시 실행하면 최신 버전으로 구동됩니다.

### 1.6 Docker 설치 방법의 장단점

#### 장점 (Docker)

* **격리된 안정적 환경:** n8n과 필요 패키지들이 컨테이너 내부에 설치되어, 호스트 Windows 환경과 분리되므로 **충돌 없이 일관된 동작**을 합니다. 한 번 설정해두면 어디서나 동일한 환경으로 재현할 수 있고, 다른 서버로 이전도 용이합니다 (이미지 이동 등).  
* **백그라운드 서비스 운용:** `-d` 옵션으로 PC 부팅 후 별도 로그인 없이도 n8n을 백그라운드 서비스처럼 실행할 수 있습니다. 필요 시 Docker Desktop 설정을 통해 Windows 시작 시 자동실행되게 하거나, 컨테이너를 재부팅시 자동 시작 설정으로 항상 가동할 수 있습니다.  
* **데이터 영속성과 쉬운 업그레이드:** 볼륨 마운트를 통해 데이터가 지속적으로 보존되며, 이미지 교체만으로 업그레이드가 **간편**합니다. 새로운 버전이 나오면 `docker pull`로 이미지 갱신 후 컨테이너만 재생성하면 업데이트 완료입니다.  
* **확장 및 배포 용이:** Docker Compose를 이용해 데이터베이스(PostgreSQL 등)나 프록시(Nginx)와 함께 손쉽게 구성할 수 있고, 운영 환경(클라우드 VM 등)으로의 배포도 수월합니다. n8n 공식 이미지에는 필요한 설정이 포함되어 있어 추가 설정이 적습니다.

#### 단점 (Docker)

* **초기 설정 복잡성:** Docker Desktop과 WSL2를 설치/구성하는 과정이 초보자에게 다소 복잡할 수 있습니다. 특히 BIOS에서 가상화 기술을 켜야 하는 등 초기 설정 허들이 있으며, Docker 자체에 익숙하지 않다면 컨테이너 개념을 이해해야 합니다.  
* **시스템 요구 사항:** Docker를 원활히 사용하려면 하드웨어 가상화 지원과 충분한 메모리가 필요합니다 (공식 권장 사양: CPU 듀얼코어 이상, RAM 2GB 이상). 오래된 PC나 가상화 미지원 환경에서는 Docker를 쓸 수 없습니다.  
* **WSL2 관련 이슈:** Windows에서 Docker는 WSL2에 의존하는데, 간혹 Hyper-V 충돌, WSL 업데이트 문제 등으로 Docker Desktop이 불안정할 수 있습니다. 예를 들어 Docker가 갑자기 시작되지 않거나 Hyper-V 관련 에러가 발생할 경우 해결에 시간이 소요될 수 있습니다. 이러한 이유로 Docker 사용에 어려움을 겪는 초보자도 있습니다.  
* **리소스 사용률:** 가상화된 컨테이너 환경이므로, 동일 작업을 직접 실행하는 것보다 메모리/디스크 등 자원을 더 소비할 수 있습니다. Docker Desktop은 백그라운드로도 리소스를 차지하므로, **경량 환경에서는 부담**이 될 수 있습니다.

*(참고: 전반적으로 Docker 방식은 초기 노력은 들지만, 장기적으로 **운영과 유지보수가 편리**하고 안정적이므로, **가능하면 Docker를 권장**하는 의견이 많습니다.)*

## 2\. Node.js 및 npm을 이용한 n8n 직접 설치

**Node.js 직접 설치 개요:** Docker를 사용하기 어려운 상황(예: BIOS에서 가상화 불가, Docker 미지원 등)에서는 **Node.js와 npm**을 통해 n8n을 직접 설치할 수 있습니다. 이 방법은 Docker 계층 없이 곧바로 호스트 OS에서 n8n을 실행하므로 초기 진입은 간단하지만, 장기적으로 수동 관리가 필요합니다. Windows에 Node.js 런타임을 설치한 후 npm으로 n8n 패키지를 실행하는 방식이며, 아래와 같이 진행합니다.

### 2.1 설치 전 준비 사항

* **Node.js 설치:** n8n은 Node.js로 개발된 애플리케이션이므로, **Node.js 런타임**이 필요합니다. LTS(Long Term Support) 버전의 Node.js를 사용하는 것이 안정적입니다 (예: Node 18.x, 20.x 등).  
    
  * **Node 설치 여부 확인:** PowerShell을 열고 `node -v`와 `npm -v`를 입력하여 버전이 출력되는지 확인합니다. 둘 다 버전 번호가 나온다면 Node.js와 npm이 이미 설치된 것입니다 (버전 예: `v20.11.0` 등). 만약 명령을 입력해도 '인식되지 않는다'는 메시지가 나온다면 Node.js를 설치해야 합니다.  
  * **Node.js 다운로드:** Node.js 공식 웹사이트에서 **Windows용 LTS 설치파일**을 다운로드합니다. 현재 시점 기준 Node 18 LTS 혹은 Node 20 LTS 버전을 권장합니다.  
  * **설치 및 환경 변수:** 다운받은 설치 프로그램을 실행하여 안내에 따라 설치합니다. 설치 시 **"PATH에 추가(Add to PATH)"** 옵션을 체크하여, 명령 프롬프트에서 `node`와 `npm`을 인식할 수 있도록 설정합니다. 설치 완료 후 **새로운 PowerShell/명령 프롬프트 창**을 열어 `node -v`로 버전이 출력되는지 재확인합니다.


* **Git (선택 사항):** n8n 설치 자체에는 필수는 아니지만, 추후 커뮤니티 노드를 설치하거나 n8n 소스 수정 등을 할 경우 Git이 필요할 수 있습니다. 미리 설치해 두면 좋습니다.  
    
* **PowerShell 실행 정책 (npx 사용시):** Windows에서 `npx`로 패키지를 실행할 때 스크립트 실행이 막히는 경우가 있습니다. 이를 방지하려면 **PowerShell 실행 정책**을 완화해야 합니다. PowerShell을 **관리자 권한**으로 열고 다음 명령을 실행하세요:  
    
  Set-ExecutionPolicy RemoteSigned \-Scope CurrentUser \-Force  
    
  위 명령은 현재 사용자에 대해 원격 서명된 스크립트 실행을 허용합니다. *(-Force 옵션은 확인 메시지 생략)*. 이 설정으로 로컬 및 신뢰된 원격 스크립트 실행이 가능해져, `npx`로 n8n 패키지를 불러와 실행하는 데 문제가 없도록 합니다.

**Why?** 기본 PowerShell 정책은 **Restricted**로 설정되어 있어, 원격에서 내려받은 스크립트의 실행이 차단될 수 있습니다. `npx`는 패키지를 내려받아 즉석에서 실행하므로, `RemoteSigned`로 정책을 변경하면 이러한 제약을 풀어줍니다. (일반적인 npm 글로벌 설치를 하는 경우에는 해당 설정이 필요없을 수도 있으나, 해두는 것을 권장)

### 2.2 n8n 패키지 설치 및 실행 (npm)

Node.js 준비가 끝났으면, npm을 이용해 n8n을 설치하거나 바로 실행할 수 있습니다. 두 가지 방법이 있습니다:

* **방법 A: 전역 설치 후 실행** – `npm install -g n8n` 명령으로 n8n을 전역 설치하고, `n8n` 명령으로 실행. (업데이트 시 다시 설치 필요)  
* **방법 B: npx로 즉시 실행** – 전역 설치 없이 `npx n8n` 명령으로 필요한 패키지를 다운로드하여 곧바로 실행. (매 실행 시 최신 버전을 가져오며, 캐시를 활용해 이후에는 빠르게 실행)

초보자에게는 \*\*방법 B(npx)\*\*를 권장합니다. 별도의 설치단계를 거치지 않아도 되고, 최신 버전으로 바로 실행되기 때문입니다. 아래에서는 npx 사용하는 법을 중심으로 설명하되, 필요에 따라 전역 설치 방법도 언급합니다.

1. **PowerShell 열기:** 일반 권한의 PowerShell 또는 명령 프롬프트를 엽니다. (관리자 권한 불필요)  
     
2. **n8n 실행 명령:** 다음 명령을 입력하고 Enter를 누릅니다:  
     
   npx n8n  
     
   첫 실행인 경우, 터미널에 **프로젝트를 설치하고 실행할지 확인**하는 메시지가 나타날 수 있습니다. `Latest version of n8n not installed. Install and run it? (Y/n)`와 같이 물어보면 `y` 키를 입력하고 Enter 쳐서 진행합니다. 그러면 npm이 n8n 최신 버전을 다운로드하여 실행합니다. (전역 설치를 이미 했다면 자동으로 그 버전을 사용할 수 있습니다.)  
     
3. **실행 로그 확인:** n8n 프로세스가 시작되면 터미널에 로그가 출력됩니다. 예를 들어 배너와 함께 "Editor is now available on port 5678" 또는 "**Webhook listening**..." 등의 메시지가 표시됩니다. 이 창을 **닫지 말고 그대로 둔 채** 다음 단계로 진행합니다. (이 창이 n8n 서버 역할을 하고 있으며, 닫으면 서버도 중지됩니다.)

**데이터 경로 및 환경:** Node.js로 직접 실행한 경우도 n8n은 기본적으로 현재 사용자 홈 디렉터리의 `.n8n` 폴더를 사용합니다. 즉 Docker 방법과 동일하게 **`C:\Users\<User>\.n8n` 폴더에 모든 데이터와 설정**이 저장됩니다. 처음 n8n을 실행하면 이 폴더가 생성되고 SQLite DB 및 암호화 키 등이 준비됩니다. 따라서 Docker \-\> Node 또는 Node \-\> Docker로 전환해도 같은 `.n8n` 경로를 마운트/사용하면 데이터가 이어집니다.

* **포트 변경:** 기본적으로 n8n은 **포트 5678**에서 HTTP 서버를 시작하며, 이는 환경 변수 `N8N_PORT`의 기본값입니다. 다른 포트를 쓰려면 n8n 실행 전에 해당 환경 변수를 지정하면 됩니다. 예를 들어 PowerShell에서는 `$Env:N8N_PORT="8080"`를 입력하고 이어서 `npx n8n`을 실행하면 8080포트로 실행됩니다. 또는 Windows **시스템 환경 변수**에 N8N\_PORT 값을 추가해 둘 수도 있습니다. (동일 PC 내의 다른 서비스와 포트 충돌 시 변경 필요)  
* **기타 환경 설정:** 이밖에 `N8N_HOST`(기본 `localhost`), `N8N_PROTOCOL`(기본 `http`), `N8N_USER_FOLDER`(기본 `"~/.n8n"`) 등의 변수를 통해 호스트명이나 데이터 폴더 위치 등을 조정할 수 있습니다. 보안 설정(예: HTTPS를 위한 `N8N_SSL_CERT`/`KEY`)도 환경 변수로 설정 가능하지만, 로컬 테스트 단계에서는 기본값으로 충분합니다.

### 2.3 실행 확인 및 접속 (Node.js 방법)

터미널 로그에 서버 시작 메시지가 나타나면, 웹 브라우저를 열고 [**http://localhost:5678**](http://localhost:5678) (또는 변경한 포트)로 접속합니다. Docker 방식과 동일하게 n8n 초기 화면이 나타나며, 최초 한 번 **계정 생성** 절차를 거치게 됩니다. 이미 Docker 방식 등으로 한 번 계정을 만들었다면 동일한 `.n8n` 데이터를 읽으므로 로그인 화면이 나올 수도 있습니다. 새로 설정하는 경우 관리자 이메일과 비밀번호를 등록하면 n8n 에디터 UI가 열립니다.

이 상태로 워크플로우를 만들고 실행해보면서 정상 작동을 확인합니다. 예를 들어 간단한 노드를 연결하여 **Execute Node**를 눌러보거나, Credentials를 추가해보는 등 테스트를 진행해볼 수 있습니다. 모든 데이터가 로컬에 저장되므로, 인터넷 연결이 안 되더라도 n8n 자체는 동작합니다.

### 2.4 n8n 프로세스 중지 및 자동 실행

n8n을 직접 실행한 터미널을 종료하려면 해당 창에서 **Ctrl \+ C** 를 누르고 confirm(확인) 질문이 나오면 `Y`를 눌러 프로세스를 종료합니다. 종료하면 localhost:5678로의 접속이 되지 않으며, 다시 사용하려면 터미널을 열고 `n8n` (혹은 `npx n8n`) 명령을 재실행해야 합니다.

로컬 개발 또는 테스트 용도로는 이처럼 **필요할 때만 수동으로 실행**해도 무방합니다. 하지만 상시 백그라운드 실행을 원한다면 다음과 같은 방법을 고려해볼 수 있습니다:

* **Windows 작업 스케줄러**를 이용해 로그인 시 `n8n` 실행 스크립트를 자동으로 실행하기  
* **윈도우 서비스**로 등록 (NSSM 같은 툴 사용)하여 n8n을 서비스로 실행하기  
* **PM2** 등의 Node 프로세스 매니저를 써서 n8n을 관리하고 자동 재시작 설정하기

다만, 이러한 설정은 초보자에게는 다소 어려울 수 있으므로 우선은 수동으로 실행/중지하며 사용해 보고, 필요한 경우 차차 적용합니다.

### 2.5 Node.js 설치 방법의 장단점

#### 장점 (Node.js 직접 실행)

* **간편한 초기 설정:** 별도 가상화 환경 없이 바로 로컬에서 실행하므로, Docker 대비 **설치 절차가 단순**합니다. Node.js만 설치되어 있다면 `npm` 명령 한두 줄로 n8n을 실행할 수 있어 가볍게 테스트해보기 좋습니다. Docker 설치나 BIOS 설정에 부담을 느끼는 사용자에게 적합한 방법입니다.  
* **시스템 자원 효율:** Docker 컨테이너 오버헤드 없이 호스트에서 직접 돌기 때문에 **메모리 및 CPU 사용량이 비교적 적을 수** 있습니다. 작은 사양의 PC에서도 무리 없이 돌릴 가능성이 높고, Docker Desktop처럼 상시 실행되는 서비스가 없어 **여유 자원 활용에 유리**합니다.  
* **파일 접근 용이:** 모든 n8n 데이터가 로컬 경로에 있으므로, 데이터베이스나 설정 파일에 직접 접근하기가 수월합니다. 필요하면 `.n8n` 폴더를 백업하거나 수정(예: `config` 파일 사용)할 수 있고, 추가적인 파일 저장/로드도 로컬 경로를 그대로 활용하면 됩니다.  
* **개발 및 커스터마이징에 유연:** Node.js로 직접 구동하므로, n8n에 커스텀 패키지를 추가한다거나 소스 코드를 수정하여 띄우는 등 **개발자 모드**로 활용하기에 적합합니다. (예: 깃허브 소스를 클론한 후 `npm run build && npm start` 등도 가능) Docker 환경보다 수정한 내용을 바로 적용하기 쉽습니다.

#### 단점 (Node.js 직접 실행)

* **수동 관리 필요:** n8n 프로세스를 사용자 세션에서 직접 돌리므로 **항상 켜두기 어려울 수** 있습니다. PC를 재부팅하면 다시 실행해줘야 하고, 자동 백그라운드 유지하려면 별도 설정이 필요합니다. 반면 Docker는 서비스처럼 동작하므로 이 부분에서 Node 직접 실행은 다소 불편합니다.  
* **업데이트 및 의존성 관리:** n8n 신규 버전이 나와도 자동으로 업데이트되지 않으며, 사용자가 직접 `npm update -g n8n` 또는 `npx n8n@latest` 등으로 업그레이드해야 합니다. 업그레이드 시 Node.js나 패키지 의존성 문제 등이 발생할 가능성도 있으며, Docker처럼 **일관된 업데이트**가 어렵습니다.  
* **환경 구성 부담:** 로컬 환경에 직접 설치하다 보니, 인증서 설정(HHTTPS)이나 프록시 설정, 포트 충돌 해결 등을 사용자가 일일이 처리해야 합니다. 반면 Docker 이미지는 기본 설정이 갖춰져 있고 독립적으로 동작하므로, Node 직접 실행은 초보자에겐 **환경 구성 난이도**가 높아질 수 있습니다. (예: SSL 적용을 위해 OpenSSL로 인증서를 만들고 env 설정하는 등의 작업)  
* **보안 및 안정성:** 호스트에서 직접 실행된 n8n은 권한 관리나 격리가 없으므로, 이론적으로 컨테이너보다 **호스트 시스템에 영향을 줄 수 있는 범위**가 큽니다. 또, 실수로 터미널을 닫거나 시스템을 종료하면 서비스가 중단되어 **신뢰성이 낮아질 수** 있습니다. 운영 환경에 바로 적용하기엔 부담이 있으며, 주로 개인용이나 개발용으로 적합합니다.  
* **커뮤니티 지원의 차이:** n8n 공식 문서나 커뮤니티에서는 Docker 사용을 기본으로 설명하는 경우가 많아, Node로 직접 설치한 경우 문제 해결 정보를 찾기 어려울 수 있습니다. 예컨대, 대부분의 튜토리얼이 Docker를 기준으로 작성되어 있어, Node 설치 사용자는 설정 파일 위치나 명령어가 달라 혼동될 수 있습니다.

## 3\. Docker vs Node.js 설치 방법 비교 및 권장 사항

두 가지 방법 모두 Windows에서 n8n을 로컬로 실행할 수 있지만, **목적과 환경에 따라 더 적합한 방법이 다릅니다.** 아래에 각 방법의 특성을 다시 한 번 비교하면 다음과 같습니다:

* **설치 난이도:** Node.js 직접 설치가 초기 단계에서는 명령어 한두 개로 끝나므로 쉬울 수 있으나, WSL2 설정만 문제없다면 Docker도 GUI 설치 마법사로 진행되어 큰 차이는 없습니다. Docker는 초기 BIOS 설정 등이 추가될 수 있습니다.  
* **환경 격리:** Docker는 컨테이너 내부에서 동작하므로 호스트와 격리되어 안전하고 일관되지만, Node 직접 실행은 호스트 환경과 섞여 돌아갑니다. 따라서 **Docker가 충돌 방지에 유리**하고, Node 방식은 **환경 영향을 받을 가능성**이 있습니다.  
* **운영 및 유지보수:** Docker는 서비스처럼 백그라운드 실행과 자동 시작, 이미지 교체로 업데이트 등 **운영 편의성**이 뛰어납니다. 반면 Node 방식은 수동 관리 요소가 많아 사용자가 관리를 잘 해야 합니다. 장기간 안정적으로 돌리기에는 Docker가 적합합니다.  
* **성능 및 자원:** Node 직접 실행은 경량으로 동작하고, Docker는 약간의 오버헤드가 있습니다. 그러나 n8n 자체가 경량 서비스이므로 **큰 성능 차이는 없습니다**. 메모리 2\~4GB 수준에서 둘 다 무난하게 실행됩니다.  
* **사용 사례:** **개인 학습/테스트** 목적이라면 간단히 Node 방식으로 시작해볼 수 있습니다. 반대로 **지속적으로 돌려야 하는 업무 자동화**를 운영한다면 Docker로 구축해두는 편이 좋습니다. Docker가 어려운 환경(예: 사내 정책으로 설치 불가)에서는 어쩔 수 없이 Node로 가되, 나중에 서버로 옮길 땐 Docker로 전환을 고려하면 됩니다.

**권장 사항:** 특별한 이유가 없다면 **Docker 설치 방법을 우선적으로 권장**합니다. 공식 가이드와 커뮤니티 조언도 Docker를 통한 셀프호스팅을 가장 안정적인 방법으로 보고 있습니다. Docker 환경을 갖추는 데 시간이 조금 걸릴 수 있지만 한 번 설정해두면 이후 관리가 수월합니다. 다만, BIOS 설정 문제 등으로 Docker 사용이 어렵다면 **Node.js 직접 실행도 충분히 가능**하며, 이 경우 보안 측면(로컬 네트워크로만 사용)과 수동 백업 등을 신경써서 사용하면 됩니다.

마지막으로, 설치 중 겪는 오류에 대한 **Quick Tips**을 정리하면 다음과 같습니다:

* **Docker 실행 오류**: "WSL2 is not supported..." 오류 시 앞서 언급한 BIOS 가상화 설정을 확인. "Port 5678 already in use" 오류 시 다른 n8n 인스턴스나 서비스 점유 가능 – 기존 컨테이너 정리(`docker stop n8n && docker rm n8n`) 후 재시도하거나 포트 번호 변경.  
* **npm 실행 오류**: `'n8n'은(는) 내부 또는 외부 명령으로...` 메시지는 PATH 문제이므로 Node.js를 재설치하며 **Add to PATH** 체크 확인. `npm install` 관련 에러는 관리자 권한 또는 네트워크 문제일 수 있으니 권한 상승 또는 네트워크 상태 확인.  
* **웹 접속 불가**: localhost:5678이 열리지 않을 때, 터미널에 에러 로그가 없는지 확인합니다. Windows 방화벽이 브라우저 접근을 막는 경우도 있으므로 방화벽 경고가 떴다면 허용해줍니다. 또한 Docker의 경우 컨테이너가 정상 실행 중인지 (`docker ps`) 체크하세요.  
* **계정 관련**: 초기 관리자 계정 생성 후 비밀번호 분실 시 `.n8n` 폴더의 `database.sqlite`를 열어 수정하기는 복잡하므로, **커뮤니티 포럼**의 비밀번호 재설정 가이드를 참고하거나 새로운 컨테이너를 생성해 연결하는 방법 등을 찾아볼 수 있습니다.

以上와 같이 Windows 환경에서 n8n을 설치하고 활용하는 두 가지 방법을 모두 살펴보았습니다. **Docker 방식**은 환경 격리와 편리한 운영이 강점이고, **Node.js 직접 방식**은 진입이 쉬워 테스트용으로 유용합니다. 자신의 환경 조건과 목적에 맞춰 적절한 방법을 선택하여 n8n의 강력한 워크플로우 자동화 기능을 활용해보세요\!
