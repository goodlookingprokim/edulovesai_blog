# `/review` 명령어 활용 가이드

`/review` 명령어가 유용한 상황들:

## 🔍 `/review` 명령어 활용 시점

### 1. **코드 완성 후 검증**
- 새로운 기능 구현 완료 시
- 복잡한 로직 작성 후
- 리팩토링 작업 완료 후
- 중요한 코드 변경 사항 발생 시

### 2. **협업 준비**
- Pull Request 생성 전
- 코드 리뷰 요청 전  
- 팀원과 공유하기 전
- 배포 전 최종 점검

### 3. **학습 목적**
- 내가 작성한 코드의 품질 확인
- 더 나은 코딩 방법 학습
- 놓친 부분 발견
- 모범 사례 적용 여부 확인

## 💡 실제 사용 예시

### 예시 1: 함수 완성 후
```
사용자: [복잡한 데이터 처리 함수 작성 완료]
사용자: /review
Claude: 코드를 검토했습니다.
✅ 좋은 점: 에러 핸들링 잘 됨
⚠️ 개선점: 성능 최적화 가능, 타입 정의 보강 필요
🔧 제안: Map 사용으로 O(n) → O(1) 개선 가능
```

### 예시 2: 컴포넌트 리뷰
```
사용자: [React 로그인 컴포넌트 작성]
사용자: /review
Claude: 컴포넌트 리뷰 결과:
✅ 상태 관리 적절함
⚠️ 접근성(a11y) 개선 필요  
🔧 제안: aria-label 추가, 키보드 내비게이션 고려
```

## 🎯 `/review`가 특히 도움되는 경우

### 초보 개발자:
- **학습 효과**: 실시간 코드 피드백
- **실수 방지**: 일반적인 버그 패턴 발견
- **모범 사례**: 더 나은 코딩 스타일 학습

### 숙련 개발자:
- **객관적 시각**: 놓친 부분 발견
- **성능 최적화**: 더 효율적인 방법 제안
- **코드 품질**: 유지보수성 개선 포인트

## 🚀 활용 패턴

### 개발 워크플로우에 통합:
```
1. 코드 작성
2. /review로 1차 검토
3. 피드백 반영하여 수정
4. 최종 완성
```

### 학습 루틴:
```
1. 새로운 기능 구현
2. /review로 분석
3. 개선점 학습 및 적용
4. 다음 개발에 활용
```

## ⭐ `/review` 사용을 잊는 이유

### 왜 안 쓰게 될까?
- **습관 부족**: 코드 완성 = 끝이라는 생각
- **시간 압박**: "빨리 다음 기능으로"
- **완벽주의**: "이미 잘 작성했다"는 착각

### 해결 방법:
- **루틴화**: 코드 완성 후 자동으로 `/review` 습관
- **품질 우선**: 빠른 개발보다 좋은 코드
- **학습 마인드**: 항상 개선할 여지가 있다고 생각

## 핵심
**`/review`는 코드 완성 후 "한 번 더 보자"는 마음가짐으로 사용하세요. 마치 글을 쓴 후 맞춤법 검사하듯, 코드도 검토가 필요합니다.**
