## Sub-Agents 개념 및 핵심 특징

### Sub-Agent란?

Sub-Agent는 Claude Code가 특정 작업을 위임할 수 있는 사전 구성된 AI 전문가들입니다. 각 Sub-Agent는:

- **전문 영역**: 특정 목적과 전문 분야를 가짐
- **독립 컨텍스트**: 메인 대화와 분리된 자체 컨텍스트 윈도우 사용
- **도구 접근 권한**: 사용할 수 있는 특정 도구들로 구성 가능
- **맞춤 시스템 프롬프트**: 행동을 안내하는 사용자 정의 시스템 프롬프트 포함

### 핵심 이점

#### 🧠 컨텍스트 보존
각 Sub-Agent는 자체 컨텍스트에서 작동하여 메인 대화의 오염을 방지합니다. 상위 목표에 집중할 수 있도록 대화 흐름을 유지합니다.

#### 🎯 전문화된 전문성
특정 도메인에 대한 세부 지침으로 미세 조정이 가능합니다. 지정된 작업에서 더 높은 성공률을 달성합니다.

#### ♻️ 재사용성
한 번 생성하면 여러 프로젝트에서 재사용이 가능합니다. 팀과 공유하여 일관된 워크플로우를 구축할 수 있습니다.

#### 🔐 유연한 권한 관리
각 Sub-Agent마다 다른 도구 접근 수준 설정이 가능합니다. 강력한 도구를 특정 Sub-Agent 유형에만 제한할 수 있습니다.

## Sub-Agent 설정 및 관리

### 빠른 시작 가이드

#### 1. Sub-Agent 인터페이스 열기
```bash
/agents
```

#### 2. 'Create New Agent' 선택
프로젝트 레벨 또는 사용자 레벨 Sub-Agent 선택

#### 3. Sub-Agent 정의
- **권장사항**: Claude로 먼저 생성한 후 개인화하여 커스터마이징
- Sub-Agent에 대한 상세 설명 및 사용 시점 기술
- 접근 권한을 부여할 도구 선택 (비워두면 모든 도구 상속)
- 인터페이스에서 사용 가능한 모든 도구를 보여주어 선택이 용이
- Claude로 생성하는 경우 `e`를 눌러 시스템 프롬프트를 편집기에서 수정 가능

#### 4. 저장 및 사용
Sub-Agent가 준비되었습니다! Claude가 적절한 때 자동으로 사용하거나 명시적으로 호출 가능:

```bash
> Use the code-reviewer subagent to check my recent changes
```

## Sub-Agent 구성 설정

### 파일 저장 위치

Sub-Agent는 YAML frontmatter가 포함된 Markdown 파일로 다음 두 위치에 저장됩니다:

| 유형 | 위치 | 범위 | 우선순위 |
|------|------|------|----------|
| 프로젝트 Sub-Agent | `.claude/agents/` | 현재 프로젝트에서만 사용 가능 | 최고 |
| 사용자 Sub-Agent | `~/.claude/agents/` | 모든 프로젝트에서 사용 가능 | 낮음 |

> 💡 **참고**: Sub-Agent 이름이 충돌할 경우, 프로젝트 레벨 Sub-Agent가 사용자 레벨 Sub-Agent보다 우선합니다.

### 파일 형식

```yaml
---
name: your-sub-agent-name
description: 이 Sub-Agent가 언제 호출되어야 하는지에 대한 설명
tools: tool1, tool2, tool3  # 선택사항 - 생략하면 모든 도구 상속
---

Sub-Agent의 시스템 프롬프트가 여기에 들어갑니다.

여러 단락으로 구성될 수 있으며 Sub-Agent의 역할, 능력, 문제 해결 접근법을 명확히 정의해야 합니다.

구체적인 지침, 모범 사례, Sub-Agent가 따라야 할 제약사항을 포함하세요.
```

### 구성 필드

| 필드 | 필수 | 설명 |
|------|------|------|
| `name` | 예 | 소문자와 하이픈을 사용한 고유 식별자 |
| `description` | 예 | Sub-Agent의 목적에 대한 자연어 설명 |
| `tools` | 아니오 | 쉼표로 구분된 특정 도구 목록. 생략하면 메인 스레드의 모든 도구 상속 |

## 도구 접근 권한

Sub-Agent는 Claude Code의 모든 내부 도구에 대한 접근 권한을 부여받을 수 있습니다.

> 💡 **권장사항**: `/agents` 명령어를 사용하여 도구 접근을 수정하세요. MCP 서버 도구를 포함한 모든 사용 가능한 도구를 나열하는 대화형 인터페이스를 제공하여 필요한 도구를 쉽게 선택할 수 있습니다.

### 도구 구성을 위한 두 가지 옵션:

#### 1. tools 필드 생략
메인 스레드의 모든 도구(MCP 도구 포함) 상속 (기본값)

#### 2. 개별 도구 지정
쉼표로 구분된 목록으로 세부 제어 (수동 편집 또는 `/agents`를 통해 편집 가능)

### MCP 도구
Sub-Agent는 구성된 MCP 서버의 MCP 도구에 접근할 수 있습니다. `tools` 필드가 생략되면 메인 스레드에서 사용 가능한 모든 MCP 도구를 상속받습니다.

## 실무 활용 사례

### `/agents` 명령어 사용 (권장)

`/agents` 명령어는 Sub-Agent 관리를 위한 종합적인 인터페이스를 제공합니다:

```bash
/agents
```

이 명령어로 열리는 대화형 메뉴에서 다음을 수행할 수 있습니다:

- 사용 가능한 모든 Sub-Agent 보기 (내장, 사용자, 프로젝트)
- 가이드된 설정으로 새 Sub-Agent 생성
- 도구 접근 권한을 포함한 기존 사용자 정의 Sub-Agent 편집
- 사용자 정의 Sub-Agent 삭제
- 중복이 있을 때 활성화된 Sub-Agent 확인
- 사용 가능한 도구의 완전한 목록으로 도구 권한을 쉽게 관리

### 직접 파일 관리

파일을 직접 작업하여 Sub-Agent를 관리할 수도 있습니다:

```bash
# 프로젝트 Sub-Agent 생성
mkdir -p .claude/agents

echo '---
name: test-runner
description: 테스트를 실행하고 실패를 수정하기 위해 적극적으로 사용
---

당신은 테스트 자동화 전문가입니다.

코드 변경사항을 발견하면 적절한 테스트를 적극적으로 실행하세요.
테스트가 실패하면 원래 테스트 의도를 보존하면서 실패를 분석하고 수정하세요.' > .claude/agents/test-runner.md

# 사용자 Sub-Agent 생성
mkdir -p ~/.claude/agents
# ... Sub-Agent 파일 생성
```

## 효과적인 Sub-Agent 사용법

### 자동 위임

Claude Code는 다음을 기반으로 작업을 적극적으로 위임합니다:

- 요청의 작업 설명
- Sub-Agent 구성의 `description` 필드
- 현재 컨텍스트와 사용 가능한 도구

> 💡 **팁**: 더 적극적인 Sub-Agent 사용을 장려하려면 `description` 필드에 "적극적으로 사용" 또는 "반드시 사용해야 함"과 같은 문구를 포함하세요.

### 명시적 호출

명령에서 특정 Sub-Agent를 언급하여 요청:

```bash
> Use the test-runner subagent to fix failing tests
> Have the code-reviewer subagent look at my recent changes
> Ask the debugger subagent to investigate this error
```

## 한국 개발 환경 특화 Sub-Agent 예제

### 1. 한국어 코드 리뷰어

```yaml
---
name: korean-code-reviewer
description: 한국어로 코드 리뷰를 수행하는 전문가. 코드 작성이나 수정 후 즉시 사용
tools: Read, Grep, Glob, Bash
---

당신은 한국 개발팀을 위한 시니어 코드 리뷰어입니다.

호출될 때:
1. git diff를 실행하여 최근 변경사항 확인
2. 수정된 파일에 집중
3. 즉시 리뷰 시작

리뷰 체크리스트:
- 코드의 가독성과 단순성
- 함수와 변수명의 적절성 (한국어 주석 허용)
- 중복 코드 제거
- 적절한 에러 처리
- 시크릿이나 API 키 노출 방지
- 입력값 검증 구현
- 테스트 커버리지 확인
- 성능 고려사항 점검

우선순위별로 피드백 제공:
- 치명적 이슈 (반드시 수정)
- 경고사항 (수정 권장)
- 제안사항 (개선 고려)

수정 방법의 구체적인 예제 포함하여 한국어로 설명
```

### 2. AI 교육 콘텐츠 생성기

```yaml
---
name: ai-education-creator
description: AI 교육 자료 생성 전문가. 강의 자료, 실습 가이드, 평가 문제 생성 시 사용
tools: Read, Write, Edit
---

당신은 AI 교육 전문가로서 한국의 AI 교육 환경에 특화되어 있습니다.

호출될 때:
1. 교육 주제와 대상 레벨 파악
2. 학습 목표 명확화
3. 체계적인 교육 자료 구성
4. 실습 중심의 콘텐츠 개발

교육 자료 생성 원칙:
- 초보자도 이해할 수 있는 쉬운 설명
- 실무에 바로 적용 가능한 예제
- 단계별 학습 가이드 제공
- 한국 기업 환경에 맞는 사례 활용
- 영어 기술 용어의 한국어 병기

결과물 형식:
- 학습 목표 및 개요
- 단계별 학습 내용
- 실습 과제 및 해답
- 참고 자료 링크
```

## 기초 활용 예시 10가지 - 파인만 기법

> 📚 **파인만 기법**: 복잡한 개념을 단순하게 설명하여 누구나 이해할 수 있도록 하는 학습법입니다. 각 예시를 '초등학생도 이해할 수 있게' 설명하고, 실제 결과를 보여드립니다.

### 1. 🍕 피자 주문하듯 간단한 웹사이트 만들기

**상황**: "웹사이트가 필요한데, 피자 주문하듯 간단하게 만들 수 없을까?"

**Sub-Agent 설정**:
```yaml
---
name: simple-web-builder
description: 초보자를 위한 간단한 웹사이트 제작자. 복잡한 것은 싫어하고 단순함을 추구
tools: Write, Edit, Read
---

당신은 웹사이트를 피자 만들듯 쉽게 만드는 전문가입니다.
복잡한 프레임워크는 사용하지 마세요. HTML, CSS, JavaScript만으로 작동하는 단순한 웹사이트를 만드세요.
```

**실행 예시**:
```bash
"simple-web-builder야, 내 카페 소개 웹사이트를 만들어줘. 카페 이름, 메뉴, 위치만 있으면 돼"
```

**예상 결과**:
- 📄 **index.html**: 깔끔한 단일 페이지 웹사이트
- 🎨 **내장 CSS**: 모바일 친화적 디자인
- ☕ **카페 정보**: 이름, 메뉴, 지도 포함
- ⏱️ **소요 시간**: 2-3분

### 2. 🧹 컴퓨터 청소하듯 코드 정리하기

**상황**: "코드가 너무 더러워서 창피해. 누가 정리 좀 해줬으면..."

**Sub-Agent 설정**:
```yaml
---
name: code-cleaner
description: 코드를 방 청소하듯 깔끔하게 정리하는 전문가
tools: Read, Edit, Grep, Glob
---

당신은 코드 청소 전문가입니다.
- 사용하지 않는 코드는 휴지통에 버리세요
- 비슷한 코드는 한곳에 모으세요  
- 이해하기 어려운 부분에는 메모(주석)를 붙이세요
- 변수명이 이상하면 알아보기 쉽게 바꾸세요
```

**실행 예시**:
```bash
"code-cleaner야, 이 프로젝트 폴더를 대청소해줘. 불필요한 파일들 정리하고 코드도 읽기 쉽게 만들어줘"
```

**예상 결과**:
- 🗑️ **불필요한 파일 삭제**: 임시파일, 중복 코드 제거
- 📝 **코드 정리**: 들여쓰기, 변수명 개선
- 💬 **주석 추가**: 이해하기 어려운 로직 설명
- 📊 **정리 리포트**: 어떤 것들을 정리했는지 요약

### 3. 🔍 셜록 홈즈처럼 버그 찾기

**상황**: "프로그램이 안 돌아가는데 어디가 문제인지 모르겠어..."

**Sub-Agent 설정**:
```yaml
---
name: bug-detective
description: 셜록 홈즈처럼 버그를 추적하는 탐정
tools: Read, Grep, Bash, Write
---

당신은 코드 탐정입니다.
1. 단서를 수집하세요 (로그, 에러 메시지)
2. 가설을 세우세요 (뭐가 문제일까?)
3. 증거를 찾으세요 (코드 검사)
4. 범인을 잡으세요 (버그 수정)
5. 사건 보고서를 작성하세요
```

**실행 예시**:
```bash
"bug-detective야, 웹사이트에서 로그인이 안 돼. 뭐가 문제인지 찾아줘"
```

## 심화 활용 예시 10가지 - 스토리텔링 기법

> 📖 **스토리텔링 기법**: 실제 현업 시나리오를 통해 Sub-Agent의 고급 활용법을 배웁니다. 각 이야기는 실제 개발팀에서 일어날 수 있는 상황을 바탕으로 합니다.

### 1. 🏢 "긴급 상황! 서버가 다운됐어요" - 위기 대응팀 구성

**배경 스토리**:
새벽 3시, 온라인 쇼핑몰 서버가 갑자기 다운됐습니다. 고객들의 불만이 빗발치고, CEO는 즉시 복구를 요구합니다. 혼자서는 감당하기 어려운 상황, Sub-Agent 응급팀을 소집합니다.

**응급팀 Sub-Agents 구성**:

```yaml
# 응급 상황 총지휘관
---
name: crisis-commander
description: 서버 장애 시 전체 복구 작업을 지휘하는 응급상황 총지휘관
tools: Read, Bash, Write, Grep
---
당신은 IT 응급상황 총지휘관입니다.
1. 상황 파악: 로그 분석으로 장애 원인 추정
2. 우선순위 결정: 핵심 서비스부터 복구
3. 팀 지휘: 다른 전문가들에게 역할 분담
4. 실시간 보고: 경영진에게 상황 보고
```

```yaml
# 로그 분석 전문가
---
name: log-analyzer
description: 서버 로그를 분석하여 장애 원인을 찾는 전문가
tools: Grep, Bash, Read
---
당신은 로그 탐정입니다.
- 에러 로그에서 장애 시점 정확히 파악
- 스택 트레이스 분석으로 문제 코드 위치 특정
- 리소스 사용량 추이 분석
- 의심스러운 패턴이나 이상 트래픽 감지
```

```yaml
# 시스템 복구 엔지니어
---
name: system-healer
description: 시스템을 빠르게 복구하는 응급의료진 같은 엔지니어
tools: Bash, Edit, Write
---
당신은 시스템 응급의료진입니다.
- 생명유지장치: 핵심 서비스 최소한으로 복구
- 응급처치: 임시 패치로 즉시 서비스 가능하게
- 안정화: 시스템 리소스 최적화
- 재발 방지: 모니터링 강화
```

**실행 시나리오**:
```bash
# 1단계: 상황 파악
"crisis-commander야, 쇼핑몰 서버가 다운됐어. 전체 상황 파악하고 복구 계획 세워줘"

# 2단계: 원인 분석  
"log-analyzer야, 최근 2시간 로그를 분석해서 장애 원인을 찾아줘"

# 3단계: 응급 복구
"system-healer야, 일단 주문 시스템만이라도 빨리 살려줘. 고객이 기다리고 있어"
```

**예상 결과 스토리**:
- ⏰ **15분 내 원인 파악**: 데이터베이스 커넥션 풀 고갈
- 🚑 **30분 내 응급 복구**: 임시 서버로 트래픽 우회
- 📊 **1시간 내 완전 복구**: 근본 원인 해결 및 모니터링 강화
- 📋 **사후 보고서**: 재발 방지를 위한 개선사항 도출

### 2. 🎯 "신입이 왔어요" - 온보딩 자동화 시스템

**배경 스토리**:
스타트업이 급성장하면서 매주 신입 개발자가 입사합니다. 매번 개발환경 설정부터 코드 리뷰까지 일일이 가르치느라 기존 개발자들이 지쳐갑니다. 이제 AI 멘토가 신입 온보딩을 담당합니다.

```yaml
당신은 신입 개발자의 든든한 멘토입니다.
1. 개인별 학습 계획 수립
2. 진도 체크 및 피드백
3. 막히는 부분 해결사
4. 팀 문화 안내
5. 정기적인 성장 리포트
```

```yaml
# 개발환경 설정 도우미

당신은 개발환경 설정 전문가입니다.
- OS별 맞춤 설정 (Windows, Mac, Linux)

- 프로젝트별 환경 구성
- 트러블슈팅 및 해결
```

```yaml
# 코드 리뷰 트레이너
---
name: code-review-trainer

당신은 신입을 위한 친절한 시니어입니다.

- 왜 그렇게 개선해야 하는지 이유 설명
- 실습 과제 제공
- 격려와 동기부여
```

**실행 시나리오**:
```bash
# Day 1: 환경 설정
"onboarding-manager야, 김신입님이 입사했어. 프론트엔드 개발자 온보딩 계획 세워줘"

# Day 3: 첫 코드 리뷰  
"code-review-trainer야, 김신입님이 첫 컴포넌트를 만들었어. 친절하게 리뷰해줘"

# Week 2: 진도 체크
"onboarding-manager야, 김신입님 2주차 진도 체크하고 피드백 보고서 만들어줘"
```

**예상 결과 스토리**:
- 📋 **개인 맞춤 계획**: 백그라운드 분석 후 4주 학습 로드맵
- 💻 **자동 환경설정**: 원클릭으로 개발환경 완성
- 📚 **단계별 미션**: 실습 위주의 학습 과제
- 📈 **성장 추적**: 주간 성과 리포트 및 개선사항

### 3. 🔐 "해킹당했어요!" - 보안 사고 대응팀

**배경 스토리**:
어느 날 아침, 보안팀에서 연락이 왔습니다. "의심스러운 활동이 감지됐습니다. 즉시 조사해주세요." 개인정보가 유출됐을 가능성이 있어 신속한 대응이 필요합니다.

- 🎨 **적응형 UI**: 사용자별 맞춤 인터페이스, 접근성 크게 개선
- 💰 **매출 영향**: 개인화 도입 후 매출 40% 증가

## Agent 워크플로우 철학

### Agent 워크플로우란?

**전통적인 LLM 사용 방식**:
- 질문 → 답변 (일회성)
- 단방향 상호작용
- 품질 개선 제한적

**Agent 워크플로우 방식**:
- 질문 → 사고 → 연구 → 수정 → 개선 (반복적)
- 자율적인 품질 향상
- 실제 팀워크와 유사한 협업

### 핵심 구성 요소

#### 1. **Reflection (반성/검토)**
- 자신의 작업 결과를 스스로 검토
- 개선점 자동 발견 및 수정
- 품질 기준에 따른 자가 평가

#### 2. **Tool Use (도구 활용)**
- 필요한 도구를 자율적으로 선택
- 작업에 맞는 권한 관리
- 효율적인 리소스 활용

#### 3. **Planning (계획 수립)**
- 작업 순서 자율 결정
- 우선순위 동적 조정
- 의존성 관계 파악

#### 4. **Multi-agent Collaboration (다중 에이전트 협업)**
- CTO, 개발자, 테스터 등 역할 분담
- 단일 LLM이 여러 전문가 역할 수행
- 실제 개발팀처럼 상호 피드백

## FAQ 및 트러블슈팅

### 일반적인 질문

**Q: Claude Code와 GitHub Copilot의 가장 큰 차이점은 무엇인가요?**
A: Claude Code는 파일을 직접 읽고 쓸 수 있으며, Agent 시스템을 통해 여러 전문가가 협업하는 방식으로 작동합니다. Copilot은 코드 완성에 특화되어 있지만, Claude Code는 전체 프로젝트 라이프사이클을 관리할 수 있습니다.

**Q: Agent를 몇 개까지 만들 수 있나요?**
A: 이론적으로는 제한이 없지만, 실제로는 5-7개 이내가 관리하기 좋습니다. 너무 많으면 역할이 중복되거나 대화가 복잡해질 수 있어요.

**Q: Sub-Agent 설정을 다른 프로젝트에서도 사용할 수 있나요?**
A: Personal Agent는 모든 프로젝트에서 사용 가능하고, Project Agent는 해당 프로젝트에서만 사용할 수 있습니다. `.claude/agents/` 폴더를 복사해서 다른 프로젝트에 적용할 수도 있어요.

### 기술적 질문

**Q: Agent 간에 데이터를 공유할 수 있나요?**
A: 네! Agent들은 같은 프로젝트 디렉토리를 공유하므로, 파일을 통해 데이터를 전달할 수 있습니다. 또한 한 Agent가 작업한 내용을 다른 Agent가 이어받아 수정하거나 확장할 수 있어요.

**Q: API 비용이 많이 나오나요?**
A: Agent를 사용하면 일반적인 대화보다 더 많은 토큰을 사용합니다. 하지만 작업 효율성이 크게 향상되기 때문에 비용 대비 효과는 충분합니다. Claude Pro 또는 API 사용량을 모니터링하며 사용하세요.

### 활용 팁

**Q: Agent 이름을 어떻게 짓는 게 좋을까요?**
A: 짧고 기억하기 쉬운 이름이 좋습니다. 예: "민호", "준호", "진수" 등. "코드작성전문가민호"처럼 긴 이름은 매번 입력하기 번거롭습니다. 또한 "민호야", "준호야" 등 친근한 호칭을 사용하면 더 자연스럽게 사용할 수 있어요.

**Q: 한 번에 여러 Agent를 동시에 호출할 수 있나요?**
A: 아니요, Claude Code는 한 번에 하나의 Agent만 활성화됩니다. 하지만 순차적으로 여러 Agent를 호출해서 마치 팀이 협업하는 것처럼 사용할 수 있습니다.

### 🔧 트러블슈팅

#### Agent 생성 관련
- **"Agent not found"**: `/agents list`로 Agent 목록 확인, 정확한 이름 입력
- **"Tool access denied"**: Agent 생성 시 필요한 도구 권한 누락, Agent 삭제 후 재생성 필요
- **"Multiple agents with same name"**: 자동으로 숫자 추가됨 (민호2, 민호3)

#### 성능 관련
- **응답 속도 저하**: 너무 많은 도구 권한 제거, Sub-Agent별 역할 명확히 분리
- **메모리 사용량 증가**: 불필요한 Agent 삭제, 컨텍스트 정리

## 🎯 최종 요약

Claude Code Sub-Agents는 개발 워크플로우를 혁신적으로 개선할 수 있는 강력한 도구입니다. 전문화된 AI 어시스턴트를 통해 작업을 효율적으로 분배하고, 컨텍스트를 보존하며, 팀 전체의 생산성을 향상시킬 수 있습니다.

### 핵심 요점
- **전문화**: 각 Agent가 특정 영역에 특화되어 높은 품질의 결과 제공
- **협업**: 실제 개발팀처럼 여러 Agent가 순차적으로 협업
- **효율성**: 컨텍스트 보존과 역할 분담으로 전체적인 효율성 향상
- **유연성**: 프로젝트와 팀의 요구사항에 맞게 커스터마이징 가능

이 가이드를 통해 Sub-Agent 시스템을 최대한 활용하여 개발 프로세스를 최적화하세요!
