# Claude Code 슬래시 명령어 문법: “명령어 카드” 만들기

## 0) 한 장면으로 시작해볼게요 (스토리)

당신은 팀에 새로 들어온 주니어 개발자입니다.  
매일 비슷한 일을 반복하죠.

- “이 질문을 더 좋은 프롬프트로 바꿔줘”
    
- “이 에러 로그 해결 방법 알려줘”
    
- “이 파일에서 특정 문자열 찾아줘”
    

그래서 선배가 말합니다.

> “그거 매번 타이핑하지 말고, `/명령어` 하나로 끝내자.  
> 그 대신 **명령어가 무엇을 하는지**, **어떤 도구를 써도 되는지**, **사용자가 뭘 입력해야 하는지**를 ‘카드’처럼 적어두면 돼.”

그 ‘카드의 앞면’이 바로 **YAML 프론트매터(frontmatter)** 입니다.

---

## 1) YAML 프론트매터는 뭐예요?

**한 줄 요약:**  
슬래시 명령어 파일의 맨 위에 붙는 “설정 라벨”입니다.

마치 **라면 봉지의 표지**처럼요.

- 이 라면은 어떤 라면인지(설명)
    
- 조리할 때 어떤 도구를 써도 되는지(가능 도구)
    
- 물을 얼마나 넣어야 하는지(입력 형식)
    

이게 프론트매터가 하는 일입니다.

### 기본 형태

```yaml
---
description: 한 문장 설명
allowed-tools: 사용할 도구들
argument-hint: 입력 형식 안내
---
```

- `---` 로 시작해서 `---` 로 끝나는 “설정 구역”
    
- 그 아래(프론트매터 이후)에 실제 명령 프롬프트/지시문이 들어갑니다
    

---

# 2) 3가지 핵심 요소 (이거만 잡으면 됩니다)

## A. `description` = “이 버튼은 뭐 하는 버튼이죠?”

**역할:**  
사용자가 `/`를 입력해서 목록을 볼 때 **설명으로 표시**됩니다.

**좋은 작성법:**

- “한 문장”으로, 결과가 무엇인지 보이게
    
- “동사 + 결과” 형태가 읽기 쉬움
    

예시:

```yaml
description: 질문을 빠르게 개선 (3버전 + 영어)
```

이 문장을 보면 사용자는 바로 이해합니다.

- “아, 이거 실행하면 질문을 개선해주고”
    
- “3가지 버전이 나오고”
    
- “영어 버전도 주는구나”
    

✅ 파인만 체크: 초등학생도 이해할 만큼 단순한가?  
→ “질문을 더 좋게 만들어주는 버튼”이면 통과입니다.

---

## B. `allowed-tools` = “이 명령어에게 허용된 ‘손’은 무엇인가?”

여기서 핵심 개념 하나만 잡으면 됩니다.

> 슬래시 명령어는 ‘일을 하는 작은 로봇’이고,  
> `allowed-tools`는 그 로봇에게 **허용된 손(도구)** 목록입니다.

**왜 필요하죠?**  
안전과 예측 가능성 때문입니다.

- 읽기만 하는 명령인데 파일을 막 써버리면 위험
    
- 터미널 실행까지 열어버리면 더 큰 영향 가능
    

그래서 “이 로봇은 읽기만 가능”, “이 로봇은 쓰기까지 가능”처럼 **권한을 제한**합니다.

### 주요 도구(사용자 관점 요약)

- `AskUserQuestion` : 사용자에게 추가 질문하기
    
- `Read` : 파일 읽기
    
- `Glob` : 파일 패턴으로 찾기(예: `src/**/*.ts`)
    
- `Grep` : 파일 내용에서 검색
    
- `Write` : 새 파일 쓰기
    
- `Edit` : 기존 파일 수정
    
- `Bash` : 터미널 명령 실행
    

### 예시 1) 안전한 명령어(읽기만)

```yaml
allowed-tools: Read, Glob, Grep
```

> “검색하고 읽어서 알려주기”는 가능  
> “파일을 바꾸는 행동”은 불가

### 예시 2) 코드 생성/수정 명령어(쓰기 허용)

```yaml
allowed-tools: Read, Write, Edit, Bash
```

> “파일 수정 + 터미널 실행”까지 가능  
> 대신 그만큼 책임도 큼(명령 설계 시 더 조심)

✅ 파인만 체크: `allowed-tools`를 한 문장으로?  
→ “이 명령어가 할 수 있는 행동의 범위를 정하는 안전장치.”

---

## C. `argument-hint` = “사용자에게 무엇을 입력받을래?”

**역할:**  
명령어 실행 시 사용자가 **어떤 형식으로 입력해야 하는지** 안내합니다.

즉, UI에 뜨는 “입력 자리표시자(placeholder)” 느낌입니다.

### 좋은 예시들

```yaml
argument-hint: [개선할 질문]
argument-hint: <file path>
argument-hint: <error message>
```

- `[]` : 자연어 입력(질문/요청)을 받는 느낌
    
- `<>` : 구조적인 값(경로, 메시지 등)이라는 느낌
    

✅ 파인만 체크: 이게 없으면?  
→ 사용자는 “뭘 넣어야 하지?”에서 멈춥니다.

---

# 3) 초보자용 핵심 정리(암기용)

슬래시 명령어 프론트매터는 딱 3줄로 생각하면 됩니다.

1. **description**: 이 버튼 뭐함? (한 문장)
    
2. **allowed-tools**: 어디까지 해도 됨? (권한)
    
3. **argument-hint**: 뭘 입력하면 됨? (입력 안내)
    

---

# 4) 실전 감각: “좋은 슬래시 명령어”의 기준

- **description은 결과를 말한다**: “~해준다”가 보이게
    
- **allowed-tools는 최소 권한**: 꼭 필요한 도구만 열기
    
- **argument-hint는 사용자 멈춤을 없앤다**: 입력 예측 가능하게
