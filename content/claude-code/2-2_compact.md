# `/compact` 명령어 효율적 사용법

`/compact` 명령어를 효율적으로 사용하는 시점들:

## 🎯 효율적 사용 시점

### 1. **대화가 길어졌을 때**
- 20-30번 이상 주고받은 대화
- 스크롤이 너무 길어진 상황
- 응답 속도가 느려진 경우

### 2. **프로젝트 단계 전환 시**
- 기획 → 개발 단계로 넘어갈 때
- 개발 → 테스트 단계로 넘어갈 때
- 한 기능 완료 후 다음 기능 시작

### 3. **정보 정리가 필요할 때**
- 여러 시행착오 후 결론 도출
- 불필요한 중간 과정들이 많을 때
- 핵심 내용만 남기고 싶을 때

## ⚡ 성능상 이점

### Before `/compact`:
```
❌ 긴 대화 기록 → 느린 응답
❌ 불필요한 세부사항들 → 컨텍스트 낭비  
❌ 중복된 정보들 → 혼란 가능성
```

### After `/compact`:
```
✅ 압축된 대화 → 빠른 응답
✅ 핵심 정보만 → 효율적 컨텍스트
✅ 정리된 내용 → 명확한 이해
```

## 🔄 실제 사용 예시

### 시나리오 1: 긴 디버깅 과정 후
```
[30번의 시행착오 대화 후]
사용자: /compact
Claude: 핵심 내용을 압축했습니다.
- 문제: CORS 에러
- 해결책: proxy 설정
- 결과: 정상 작동

[이제 간결한 컨텍스트로 다음 작업 진행]
```

### 시나리오 2: 프로젝트 단계 변경
```
[기획 단계 완료 후]
사용자: /compact
Claude: 기획 단계 결과를 정리했습니다.
- 기술 스택: React + TypeScript
- 주요 기능: 로그인, 대시보드, 설정
- 다음 단계: 개발 시작

사용자: 이제 로그인 컴포넌트부터 만들어보자
```

### 시나리오 3: 지시사항이 포함된 압축
```
[긴 디버깅 과정 후]
사용자: /compact 문제점과 최종 해결책만 남겨줘. 중간에 시도했던 실패한 방법들은 제거해.
Claude: 요청대로 정리했습니다.
- 문제: CORS 에러 발생
- 최종 해결책: proxy 설정 추가
- 결과: API 호출 정상 작동

[실패한 시도들(JWT 토큰 문제, 헤더 설정 등)은 제거됨]
```

## 📝 `/compact [instructions]` 고급 활용

### 상황별 지시사항 예시:
```
/compact 결정된 기술 스택과 아키텍처 정보만 남겨줘
/compact 완료된 작업 목록과 다음 할 일만 정리해줘  
/compact 버그 수정 과정은 생략하고 최종 코드만 보여줘
```

### 장점:
- **맞춤형 압축**: 원하는 내용만 선별적으로 보존
- **정확한 정리**: 구체적인 지시로 더 나은 결과
- **효율적 관리**: 필요한 정보에 집중 가능

**팁**: 단순히 `/compact`만 사용하는 것보다 구체적인 지시사항을 주면 원하는 방향으로 더 정확한 압축이 가능합니다.

## 🚫 `/clear`와의 차이점

### `/compact`:
- ✅ 중요 정보 보존
- ✅ 컨텍스트 유지
- ✅ 성능 개선
- ✅ 안전한 정리

### `/clear`:
- ❌ 모든 정보 삭제
- ❌ 컨텍스트 완전 손실
- ❌ 처음부터 다시 시작
- ❌ 복구 불가능

## 💡 사용 타이밍 팁

### 좋은 타이밍:
- **문제 해결 완료 직후**
- **중간 점검이 필요할 때**
- **새로운 주제 시작 전**
- **대화가 복잡해졌을 때**

### 피해야 할 타이밍:
- **진행 중인 작업 도중**
- **여러 선택지 검토 중**
- **아직 결론이 안 난 상황**

## 🎨 활용 패턴

### 주기적 사용:
```
1. 작업 시작 → 진행 → /compact
2. 다음 작업 → 진행 → /compact
3. 반복...
```

### 문제 해결 후:
```
1. 문제 발생 → 여러 시도 → 해결
2. /compact으로 핵심만 정리
3. 다음 작업에 집중
```

## 핵심
**`/compact`는 대화를 지우지 않고 압축하는 스마트한 정리 도구입니다. `/clear`보다 안전하면서도 성능을 개선할 수 있어 자주 사용해도 좋습니다.**
