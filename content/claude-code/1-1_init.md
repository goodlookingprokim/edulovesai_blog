# `/init` 명령어 사용 가이드

`/init` 명령어를 사용할 경우와 사용하지 않을 경우를 비교해서 설명합니다.

## `/init` 명령어 사용할 경우

### 상황:
- 새로운 프로젝트 시작
- 기존 프로젝트를 Claude에게 소개
- 복잡한 코드베이스 작업
- 팀 프로젝트 협업

### 장점:
```
✅ 프로젝트 구조 자동 분석
✅ 관련 파일들 컨텍스트에 추가
✅ 프로젝트 설정 파일 인식 (package.json, requirements.txt 등)
✅ 일관된 코딩 스타일 유지
✅ 프로젝트 전체적인 이해도 향상
```

### 예시:
```
사용자: /init
Claude: 프로젝트를 분석하고 있습니다...
- React 프로젝트 감지
- TypeScript 설정 확인
- 컴포넌트 구조 파악
- 테스트 환경 인식
준비 완료! 이제 프로젝트에 대해 질문하거나 작업을 요청하세요.
```

## `/init` 명령어 사용하지 않을 경우

### 상황:
- 단순한 일회성 질문
- 코드 스니펫만 다루는 경우
- 프로젝트와 무관한 일반적인 프로그래밍 질문
- 빠른 디버깅이나 문법 확인

### 특징:
```
⚠️ 파일별 개별 처리
⚠️ 프로젝트 전체 컨텍스트 부족
⚠️ 일관성 부족 가능성
⚠️ 매번 파일 구조 설명 필요
```

### 예시:
```
사용자: 이 함수 어떻게 최적화할 수 있을까요? [코드 첨부]
Claude: 해당 함수만 보고 최적화 제안
(다른 파일과의 연관성이나 프로젝트 패턴 고려 불가)
```

## 실제 비교 예시

### 시나리오: React 컴포넌트 생성 요청

**`/init` 사용한 경우:**
```
사용자: /init
Claude: [프로젝트 분석 완료]

사용자: 로그인 컴포넌트 만들어줘
Claude: 
- 기존 컴포넌트 스타일 맞춤
- TypeScript 인터페이스 활용
- 프로젝트의 상태관리 패턴 적용
- 기존 CSS 모듈 구조 따름
- 테스트 파일도 함께 생성
```

**`/init` 사용하지 않은 경우:**
```
사용자: 로그인 컴포넌트 만들어줘
Claude:
- 일반적인 React 컴포넌트 생성
- 프로젝트 구조 모름
- 스타일링 방식 불확실
- 상태관리 방식 모름
- 추가 설명 필요
```

## 권장 사용 시점

### `/init` 사용 권장:
- 프로젝트 첫 작업 시
- 복잡한 기능 추가/수정
- 다중 파일 작업
- 아키텍처 관련 작업

### `/init` 불필요:
- 단순 문법 질문
- 알고리즘 문제 해결
- 일반적인 프로그래밍 개념 질문
- 코드 스니펫 리뷰

## 코드 스니펫(Code Snippet)이란?

**코드 스니펫**은 작은 코드 조각을 말합니다.

### 간단한 설명
- **한국어**: 코드 조각, 코드 토막
- **의미**: 완전한 프로그램이 아닌 일부분의 코드
- **크기**: 보통 몇 줄에서 십여 줄 정도

### 예시들

#### 함수 하나
```javascript
function add(a, b) {
  return a + b;
}
```

#### 조건문
```python
if user_age >= 18:
    print("성인입니다")
```

#### HTML 태그
```html
<button onclick="alert('클릭!')">버튼</button>
```

### 일상 비유
- **요리**: 전체 레시피가 아닌 "양파 볶는 방법"만
- **음악**: 전체 곡이 아닌 "후렴구"만
- **글**: 전체 소설이 아닌 "한 문단"만

### 언제 사용?
- 특정 기능만 보여줄 때
- 문제 부분만 질문할 때
- 예시로 설명할 때
- 빠른 테스트할 때

**핵심**: 전체가 아닌 **필요한 부분만** 떼어낸 코드입니다.

## 결론

`/init`는 **프로젝트 맥락이 중요한 작업**에서는 필수이고, **일반적인 프로그래밍 질문**에서는 불필요합니다. 프로젝트 작업 시 초기에 한 번 사용하면 이후 모든 작업의 품질이 향상됩니다.
