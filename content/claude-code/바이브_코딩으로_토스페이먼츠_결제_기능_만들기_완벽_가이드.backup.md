## 이야기로 시작하기 - 개발자 없이 결제 시스템을 만든다고?

### 🎭 민수의 이야기

민수는 수제 키링을 파는 작은 온라인 샵을 운영하고 있습니다.
지금까지는 카카오톡으로 주문을 받고 계좌이체로 입금받았는데, 손님들이 불편해합니다.

> "결제 시스템 좀 만들어주세요!"

민수는 검색해봤습니다. 웹 개발자를 고용하면 최소 수백만 원, 시간도 몇 주가 걸립니다.
포기하려던 순간, **AI 코딩 도구**를 발견합니다.

**"코딩을 모르는 내가 결제 시스템을 만들 수 있을까?"**

### 🎯 이 가이드가 보여줄 것

이 가이드는 민수처럼 **코딩 경험이 전혀 없는 분들**도:
- AI 도구(Cursor)를 사용하여
- 실제 작동하는 결제 시스템을 만들고
- 발생하는 모든 에러를 해결하는 방법을

**처음부터 끝까지** 보여드립니다.

### 💪 여러분이 얻게 될 능력

이 가이드를 마치면:
- ✅ 토스 페이먼츠 결제 시스템 구축
- ✅ 결제 정보를 구글 시트에 자동 저장
- ✅ 개발 중 발생하는 에러 스스로 해결
- ✅ AI에게 효과적으로 질문하는 방법

**걱정하지 마세요!** 에러는 당연히 발생합니다.
이 가이드는 에러 해결 과정까지 모두 담고 있습니다.

## 바이브 코딩이란 무엇인가?

### 🎨 바이브 코딩의 정의

**바이브 코딩(Vibe Coding)**은:
- **"분위기로 코딩한다"**는 의미
- 정확한 문법을 모르고
- 깊은 원리를 몰라도
- **AI에게 원하는 것을 설명**하면
- AI가 알아서 코드를 작성해주는 개발 방식

### 🎪 요리로 이해하는 바이브 코딩

#### 전통적인 개발 (셰프가 되는 과정)
```
1단계: 요리학교 4년 다니기
  → 칼질, 불 조절, 양념 비율 모두 배우기

2단계: 레시피 정확히 외우기
  → "소금 5g, 설탕 10g, 물 200ml..."

3단계: 직접 요리하기
  → 실수하면 처음부터 다시
```

#### 바이브 코딩 (AI 셰프 활용)
```
당신: "매운 떡볶이 만들어줘,
      근데 약간 달콤하게,
      치즈 추가하고"

AI: "알겠습니다!"
    [완벽한 레시피 제시]
    [요리 과정 설명]

당신: "어, 너무 매워. 덜 맵게"

AI: "고추장을 50% 줄였습니다!"
```

### 🔑 바이브 코딩의 핵심

**핵심 1: 무엇을 원하는지만 알면 됨**
```
❌ 알아야 할 것:
   - 프로그래밍 문법
   - 데이터베이스 구조
   - API 통신 방법

✅ 알아야 할 것:
   - 내가 원하는 기능이 뭔지
   - 문제가 뭔지 설명하기
   - 에러 메시지 복사 붙여넣기
```

**핵심 2: AI가 실수하면 다시 요청**
```
시도 1: "결제 기능 만들어줘"
결과: 뭔가 부족함

시도 2: "결제 위젯도 보이게 해줘"
결과: 위젯은 보이는데 작동 안 함

시도 3: "이 샘플 코드 참고해서 만들어줘"
결과: 성공! ✅
```

**핵심 3: 에러는 성장의 계단**
```
에러 없음 → 1층
   ↓
첫 번째 에러 해결 → 2층
   ↓
두 번째 에러 해결 → 3층
   ↓
세 번째 에러 해결 → 옥상 (완성!)
```

### 🎯 왜 바이브 코딩이 가능한가?

**2020년 이전:**
- AI: "결제 기능? 무슨 말이야?"
- ❌ 불가능

**2024년 이후:**
- AI: "토스 페이먼츠 React 연동이군요!"
- AI: "샘플 코드 분석 완료"
- AI: "CORS 설정 추가했습니다"
- ✅ 가능!

최신 AI는:
- 수백만 개의 코드를 학습했고
- 개발 문서를 이해할 수 있고
- 에러 메시지를 분석해서 해결책을 제시합니다

## 웹 서비스의 기본 구조 이해하기

### 🏢 실생활로 이해하는 클라이언트-서버

#### 네이버 메일을 확인하는 과정

```
[당신의 컴퓨터]              [네이버 본사 서버]
      ↓                            ↓
  웹 브라우저 열기          메일 데이터 저장소
      ↓                            ↓
  "메일 보여줘!" ───────────→ "누구의 메일?"
      ↓                            ↓
  로그인 정보 전송 ─────────→ 본인 확인
      ↓                            ↓
  메일 화면 표시 ←───────── 메일 데이터 전송
```

### 🎭 식당으로 이해하기

**클라이언트 (Client) = 손님**
- 역할: 메뉴판을 보고 주문하기
- 위치: 당신의 컴퓨터 (웹 브라우저)
- 특징: 예쁜 화면으로 정보를 보여줌

```
클라이언트 화면:
┌─────────────────────┐
│  🛒 장바구니         │
│                     │
│  키링 A  ...  5,000원│
│  키링 B  ...  7,000원│
│                     │
│  [결제하기 버튼]     │
└─────────────────────┘
```

**서버 (Server) = 주방**
- 역할: 주문을 받아서 처리하기
- 위치: 멀리 떨어진 회사의 컴퓨터
- 특징: 실제 데이터를 저장하고 처리함

```
서버의 작업:
1. 결제 요청 받기
2. 토스 페이먼츠에 결제 처리 요청
3. 결제 완료 확인
4. 구글 시트에 주문 정보 저장
5. 결과를 클라이언트에 전달
```

### 🚗 통신 과정 상세 설명

#### 레벨 1: 기본 대화

```
손님(클라이언트): "메뉴판 주세요"
  ↓ [인터넷 통신]
주방(서버): "여기 메뉴판입니다"
  ↓ [인터넷 통신]
손님: "화면에 메뉴 표시"
```

#### 레벨 2: 결제 과정

```
손님: "키링 2개 주문하고 결제할게요"
  ↓
주방: "총 12,000원입니다. 결제 진행하시겠어요?"
  ↓
손님: "네" → [토스 결제 창 열림]
  ↓
결제 회사: "카드 정보 확인... OK!"
  ↓
주방: "결제 완료! 구글 시트에 저장했어요"
  ↓
손님: "결제 완료 화면 표시"
```

#### 레벨 3: 실제 코드 구조

```
프로젝트 폴더
│
├── client/ (클라이언트 - 손님)
│   ├── 화면 디자인 파일들
│   ├── 결제 버튼 클릭 처리
│   └── 결제 위젯 표시
│
├── server/ (서버 - 주방)
│   ├── 클라이언트 요청 받기
│   ├── 토스 페이먼츠 통신
│   ├── 결제 승인 처리
│   └── 구글 시트 저장
│
└── .env (환경 변수 - 비밀 정보)
    ├── 토스 API 키
    ├── 구글 서비스 계정 키
    └── 포트 번호
```

### 💡 왜 클라이언트와 서버를 나누나요?

**보안 때문입니다!**

```
❌ 나쁜 예: 모든 걸 클라이언트에서 처리

클라이언트 코드에 이렇게 쓰면:
  const API_KEY = "sk_test_비밀키123..."

문제점:
  → 웹사이트 방문자가 개발자 도구로 코드를 보면
  → API 키가 그대로 노출됨!
  → 누구나 내 키로 결제를 시도할 수 있음!
```

```
✅ 좋은 예: 비밀 정보는 서버에만

클라이언트: "결제 해주세요!" (API 키 모름)
  ↓
서버: 비밀 키를 사용해서 결제 처리 (안전!)
  ↓
클라이언트: "완료되었습니다!" 메시지만 받음
```

### 🎯 이 실습에서 만들 구조

```

       ↓
  1. 상품 선택

## 필요한 준비물과 도구들

### 🧰 도구 상자 준비하기

#### 1️⃣ Cursor (커서) - AI 코딩 도구

**무엇인가요?**
- AI가 탑재된 코드 편집기
- ChatGPT처럼 대화하면서 코드 작성
- 에러를 알아서 고쳐줌

**비유:**
```
일반 텍스트 편집기 = 빈 종이 + 연필
  → 모든 걸 직접 써야 함

Cursor = 똑똑한 비서가 있는 사무실
  → "이렇게 써줘" 하면 알아서 작성
  → "이 부분 틀렸어" 하면 알아서 수정
```

**설치 방법:**
```
1. cursor.sh 웹사이트 방문
2. 다운로드 버튼 클릭
3. 설치 파일 실행
4. 계정 가입 (무료)
```

**기본 사용법:**
```
1. 프로젝트 폴더 열기
2. Cmd+L (Mac) 또는 Ctrl+L (Windows)
   → AI 채팅창 열림
3. 원하는 것 설명
4. AI가 코드 작성
5. Accept 버튼으로 적용
```

#### 2️⃣ Rubber (러버) - UI 디자인 도구

**무엇인가요?**
- 노코드 웹 디자인 도구
- 드래그 앤 드롭으로 예쁜 화면 제작

**이번 실습에서:**
- 이미 만들어진 "키링샵" 디자인 사용
- 결제 버튼, 장바구니 등이 이미 완성됨

#### 3️⃣ 토스 페이먼츠 - 결제 서비스

**무엇인가요?**
- 한국의 온라인 결제 서비스
- 카드, 계좌이체, 간편결제 지원

**준비물:**
```
1. 토스 페이먼츠 개발자 사이트 가입
   → docs.tosspayments.com

2. 테스트 API 키 발급
   - 클라이언트 키: test_ck_xxxxx
   - 시크릿 키: test_sk_xxxxx

3. ⚠️ 주의:
   - 'test'가 들어간 키 = 가상 결제 (돈 안 나감)
   - 실제 키 = 진짜 결제 (돈 나감!)
```

#### 4️⃣ 구글 시트 - 간이 데이터베이스

**무엇인가요?**
- 엑셀처럼 생긴 구글 스프레드시트
- 결제 정보를 저장하는 데이터베이스로 활용

**준비 과정:**

**STEP 1: 구글 시트 생성**
```
1. Google Sheets 접속
2. 새 스프레드시트 만들기
3. 이름: "키링샵_결제정보"
4. 시트명: "결제 정보"
```

**STEP 2: 데이터 구조 설정**
```
첫 번째 행에 헤더 입력:
┌──────────┬──────────┬──────────┬──────────┬──────────┐
│ 결제일시  │ 주문번호  │ 구매자명  │ 이메일    │ 전화번호  │
├──────────┼──────────┼──────────┼──────────┼──────────┤
│ 결제금액  │ 결제방법  │ 상품명    │ 수량      │ 배송주소  │
└──────────┴──────────┴──────────┴──────────┴──────────┘
```

**STEP 3: 서비스 계정 만들기**
```
1. Google Cloud Console 접속
2. 새 프로젝트 생성: "키링샵"
3. API 및 서비스 → 사용자 인증 정보
4. 서비스 계정 만들기
   - 이름: keyring-shop-service
   - 역할: 편집자
5. 키 생성 → JSON 다운로드
   - 파일명: service-account-key.json
```

**STEP 4: 시트 공유**
```
1. 구글 시트 열기
2. 우측 상단 "공유" 버튼
3. 서비스 계정 이메일 추가
   (형식: xxxx@yyyy.iam.gserviceaccount.com)
4. 권한: 편집자
5. 완료
```

### 📦 Node.js 와 npm

**무엇인가요?**
- Node.js: 서버를 구동하는 프로그램
- npm: 필요한 코드 패키지를 설치하는 도구

**비유:**
```
Node.js = 자동차 엔진
npm = 부품 가게

"결제 기능 필요해요"
  → npm install toss-payments
  → 토스 페이먼츠 코드 다운로드
  → 프로젝트에서 사용 가능
```

**설치:**
```
1. nodejs.org 방문
2. LTS 버전 다운로드 (안정 버전)
3. 설치 후 터미널에서 확인:
   node --version
   npm --version
```

### 🛠️ 스스로 해결 가능한 에러: 신택스 에러

#### 신택스 에러란?

```
Syntax Error = 문법 오류

한국어 비유:
"나는 학교에 갑니다" ✅
"나는 학교를 갑니다" ❌ (조사 오류)

코드 비유:
const name = "홍길동"  ✅
const name = "홍길동   ❌ (따옴표 안 닫음)
```

#### 흔한 신택스 에러 유형

**1. 복사 붙여넣기 실수**
```
의도:
const API_KEY = "test_ck_123456"

실제로 붙여넣은 것:
const API_KEY = "test_ck_123456"V

문제:
→ 마지막에 'V' 추가됨 (Cmd+V의 V)
→ SyntaxError: Unexpected identifier 'V'
```

**2. 괄호 불일치**
```
❌ 틀린 코드:
function calculate() {
  return 5 + 3

// 닫는 괄호 } 빠짐!

✅ 올바른 코드:
function calculate() {
  return 5 + 3
}
```

**3. 대소문자 오류**
```
JavaScript는 대소문자를 구분합니다!

❌ 틀림:
const myname = "홍길동"
console.log(myName)  // myName은 정의 안 됨!

✅ 맞음:
const myName = "홍길동"
console.log(myName)
```

**4. 세미콜론 누락 (선택적)**
```
// 요즘은 세미콜론 없이도 작동함
const a = 5
const b = 10

// 하지만 한 줄에 여러 명령 시 필요
const a = 5; const b = 10
```

#### 신택스 에러 해결 방법

**STEP 1: 에러 메시지 읽기**
```
SyntaxError: Unexpected token '}' in JSON at position 45
                                              ↑
                                         여기 주목!

→ 45번째 위치에 문제
→ JSON 형식 오류
→ 중괄호 } 가 예상치 못한 곳에 있음
```

**STEP 2: 줄 번호 확인**
```
에러 메시지:
  at checkout.jsx:127:5
              ↑   ↑
         파일명  줄번호

1. checkout.jsx 파일 열기
2. 127번째 줄로 이동 (Ctrl+G)
3. 해당 줄 및 앞뒤 확인
```

**STEP 3: 직접 수정**
```
127번째 줄:
const data = response.json())
                           ↑
                    괄호 하나 더 많음!

수정:
const data = response.json()
```

### 🎯 실전 에러 해결 사례

#### 사례 1: "require is not defined"

**에러 발생:**
```

ReferenceError: require is not defined in ES module scope
```

**원인:**
```
ES6 모듈 시스템에서 require() 사용 불가

❌ 구식 방식:
const express = require('express')

import express from 'express'

```
Cursor에게:

Cursor가 자동으로:
1. package.json에 "type": "module" 추가
2. 모든 require를 import로 변경
3. module.exports를 export로 변경
```

```

## 성공의 열쇠 - 완벽한 프롬프트 작성법

### 📝 프롬프트의 중요성

#### 간단한 프롬프트 vs 상세한 프롬프트

**테스트: 같은 요청, 다른 결과**

```
프롬프트 A (간단):
"결제 기능 만들어줘"

결과:
→ 기본 구조만 생성
→ 에러 10개 발생
→ 결제 위젯 안 보임
→ 해결에 2시간 소요

─────────────────────────────

프롬프트 B (상세):
"@개발문서 @샘플코드 참고해서
 토스 결제 위젯 연동해 줘.
 CORS 설정하고, 포트는 3000/4000번 사용.
 .env에 API 키 저장하고..."

결과:
→ 완전한 코드 생성
→ 에러 1-2개만 발생
→ 결제 위젯 정상 작동
→ 해결에 30분 소요
```

### 🎯 완벽한 프롬프트의 6가지 요소

#### 1️⃣ 맥락 (Context)

**무엇을:**
- 현재 프로젝트 상황
- 이미 완성된 부분
- 사용 중인 기술

**예시:**
```

### 🎯 왜 결제 정보를 저장해야 하나?

#### 실제 비즈니스 시나리오

```
고객이 키링 2개를 결제했습니다.
┌────────────────────────────┐
│ 주문번호: ORD-20241012-001  │
│ 구매자: 김민수              │
│ 전화번호: 010-1234-5678     │

│ 금액: 12,000원             │
│ 배송지: 서울시 강남구...    │
└────────────────────────────┘

이 정보가 없으면:
❌ 누구에게 보내야 하는지 모름
❌ 무엇을 보내야 하는지 모름
❌ 얼마를 받았는지 확인 불가
❌ 고객 연락 불가

이 정보를 저장하면:
✅ 주문 관리 가능
✅ 배송 처리 가능
✅ 고객 문의 대응 가능
✅ 매출 분석 가능
```

### 📊 구글 시트를 DB로 사용하는 이유

#### 실제 데이터베이스 vs 구글 시트

**실제 DB (MySQL, PostgreSQL 등)**
```
장점:
✅ 빠름
✅ 안전함
✅ 대용량 처리 가능

단점:
❌ 설치/설정 복잡
❌ 서버 관리 필요
❌ 비용 발생
❌ 초보자에게 어려움
```

**구글 시트**
```
장점:
✅ 설정 초간단
✅ 무료
✅ 엑셀처럼 직관적
✅ 팀원과 공유 쉬움
✅ 직접 수정 가능

단점:
❌ 속도 느림 (대용량 시)
❌ 동시 접속 제한
❌ 프로페셔널하지 않음

결론:
초기 프로토타입이나
소규모 비즈니스에는 완벽!
```

### 🔐 구글 서비스 계정 설정

#### 서비스 계정이란?

```
일반 계정 vs 서비스 계정

일반 구글 계정:
사람: kim@gmail.com
용도: 사람이 직접 로그인해서 사용

서비스 계정:
프로그램: robot@project.iam.gserviceaccount.com
용도: 프로그램이 자동으로 접근

비유:
일반 계정 = 정규 직원 (출퇴근 카드)
서비스 계정 = 배달 로봇 (특별 출입증)
```

#### 서비스 계정 생성 단계

**STEP 1: Google Cloud Console 접속**
```
1. console.cloud.google.com 접속
2. 구글 계정으로 로그인
3. "프로젝트 선택" 클릭
4. "새 프로젝트" 클릭

```
프로젝트 이름: keyring-shop-project
위치: 조직 없음

생성 클릭!

**STEP 3: Google Sheets API 활성화**
```
1. 좌측 메뉴 → "API 및 서비스" → "라이브러리"
2. 검색: "Google Sheets API"
3. 클릭 → "사용 설정" 버튼
```

**STEP 4: 서비스 계정 만들기**
```
1. 좌측 메뉴 → "API 및 서비스" → "사용자 인증 정보"
2. 상단 "+ 사용자 인증 정보 만들기"
3. "서비스 계정" 선택

서비스 계정 세부정보:
- 이름: keyring-shop-service
- ID: keyring-shop-service (자동 생성)
- 설명: "키링샵 결제 정보 저장용"

역할 부여:
- 역할: "편집자" 선택

완료!
```

**STEP 5: JSON 키 다운로드**
```
1. 생성된 서비스 계정 클릭
2. "키" 탭 선택
3. "키 추가" → "새 키 만들기"
4. "JSON" 선택 → "만들기"

다운로드됨:
keyring-shop-project-xxxxx.json

이 파일을:
1. 프로젝트 루트 폴더에 복사
2. 이름 변경: service-account-key.json
3. ⚠️ 절대 Git에 업로드하지 말 것!
```

**STEP 6: 구글 시트 공유**
```
1. 다운로드한 JSON 파일 열기
2. "client_email" 항목 찾기:
   "keyring-shop-service@xxxxx.iam.gserviceaccount.com"
   이 이메일 복사!

3. 구글 시트 열기
4. 우측 상단 "공유" 버튼
5. 복사한 이메일 붙여넣기
6. 권한: "편집자"
7. "보내기" (알림 메일 안 보내도 됨)

완료!
```

**2. 구글 시트 모듈 (server/utils/googleSheets.js)**
```javascript
import { google } from 'googleapis';
import fs from 'fs';
import path from 'path';

// 서비스 계정 키 파일 로드
const keyFile = path.join(process.cwd(), '../service-account-key.json');
const keys = JSON.parse(fs.readFileSync(keyFile, 'utf8'));

// 구글 시트 API 인증
const auth = new google.auth.GoogleAuth({
  credentials: keys,
  scopes: ['https://www.googleapis.com/auth/spreadsheets'],
});

// 시트에 데이터 추가하는 함수
export async function appendToSheet(data) {
  try {
    const sheets = google.sheets({ version: 'v4', auth });
    const spreadsheetId = process.env.GOOGLE_SHEET_ID;

    // 데이터를 배열 형태로 변환
    const values = [[
      data.결제일시,
      data.주문번호,
      data.구매자명,
      data.이메일,
      data.전화번호,
      data.결제금액,
      data.결제방법,
      data.상품명,
      data.수량,
      data.배송주소,
      data.토스거래ID,
      data.결제상태
    ]];

    // 시트에 추가
    const response = await sheets.spreadsheets.values.append({
      spreadsheetId,
      range: '결제 정보!A:L',
      valueInputOption: 'USER_ENTERED',
      requestBody: { values },
    });

    console.log('✅ 구글 시트 저장 성공:', response.data);
    return response.data;

  } catch (error) {
    console.error('❌ 구글 시트 저장 실패:', error);
    throw error;
  }
}
```

**3. 결제 API에서 호출 (server/routes/payment.js)**
```javascript
import { appendToSheet } from '../utils/googleSheets.js';

// 결제 승인 엔드포인트
app.post('/api/payment/confirm', async (req, res) => {
  try {
    // 1. 토스 페이먼츠에 결제 승인 요청
    const paymentResult = await confirmPayment(req.body);

    // 2. 결제 성공 시 구글 시트에 저장
    if (paymentResult.status === 'DONE') {
      const sheetData = {
        결제일시: new Date().toLocaleString('ko-KR'),
        주문번호: paymentResult.orderId,
        구매자명: req.body.customerName,
        이메일: req.body.customerEmail,
        전화번호: req.body.customerPhone,
        결제금액: paymentResult.totalAmount,
        결제방법: paymentResult.method,
        상품명: req.body.orderName,
        수량: req.body.quantity,
        배송주소: req.body.address,
        토스거래ID: paymentResult.paymentKey,
        결제상태: '완료'
      };

      await appendToSheet(sheetData);
    }

    // 3. 클라이언트에 결과 전송
    res.json({ success: true, data: paymentResult });

  } catch (error) {
    console.error('결제 처리 에러:', error);
    res.status(500).json({ success: false, error: error.message });
  }
});
```

### 🐛 자주 발생하는 문제

#### 문제 1: 시트에 저장 안 됨

**증상:**
```
결제는 완료되는데 구글 시트에 데이터 없음
```

**확인사항:**
```
1. 서비스 계정 이메일로 시트 공유했는지?
   → 시트 우측 상단 "공유" 확인

2. 환경 변수가 server 폴더의 .env에 있는지?
   → ❌ 루트/.env (X)
   → ✅ server/.env (O)

3. 시트 ID가 정확한지?
   → URL에서 /d/와 /edit 사이의 긴 문자열

4. 서버를 재시작했는지?
   → 환경 변수 변경 시 재시작 필수!
```

#### 문제 2: 권한 에러

**증상:**
```
Error: The caller does not have permission
```

**해결:**
```
1. Google Cloud Console에서
   Google Sheets API가 활성화되었는지 확인

2. 서비스 계정에 "편집자" 역할 부여

3. 구글 시트 공유 설정에서
   서비스 계정 이메일이 "편집자" 권한인지 확인
```

#### 문제 3: 파일 경로 에러

**증상:**
```
Error: ENOENT: no such file or directory 'service-account-key.json'
```

**해결:**
```
1. 파일 위치 확인:
   프로젝트루트/service-account-key.json

2. 환경 변수 경로 확인:
   server/.env:
   GOOGLE_SERVICE_ACCOUNT_KEY_PATH=../service-account-key.json

   (../ = 상위 폴더)
```

**해결 방법 A: 프로세스 종료 (Mac)**

```bash
# 1. 포트 사용 중인 프로세스 찾기

# 결과:

kill -9 12345

```bash
# 1. 포트 사용 중인 프로세스 찾기

# 결과:
# TCP    0.0.0.0:4000    0.0.0.0:0    LISTENING    12345

taskkill /PID 12345 /F

npm start
```

#### 4️⃣ CORS 에러 💥

**증상:**
```
브라우저 콘솔:
Access to fetch at 'http://localhost:4000/api/payment'
from origin 'http://localhost:3000' has been blocked
by CORS policy
```

**쉽게 이해하기:**
```
CORS = 출입국 심사

상황:
당신(클라이언트:3000)이
은행(서버:4000)에 가서
돈을 인출하려 함

CORS 에러 = 은행이 거부:
"당신 신분증 없잖아요?
 우리 은행은 승인된 사람만 받아요!"

해결:
은행(서버)에서 설정:
"localhost:3000에서 오는 요청은 허용"
```

**해결: 서버에 CORS 설정 추가**

```javascript
// server/index.js

import cors from 'cors';

const app = express();

// CORS 설정
app.use(cors({
  origin: 'http://localhost:3000',  // 클라이언트 주소
  credentials: true,                // 쿠키 허용
}));

// 또는 환경 변수 사용
app.use(cors({
  origin: process.env.CLIENT_URL,
  credentials: true,
}));
```

**체크리스트:**
```
□ cors 패키지 설치했나?
  npm install cors

□ app.use(cors())를 추가했나?
  → 다른 미들웨어보다 위에 배치!

□ origin 주소가 정확한가?
  ✅ http://localhost:3000
  ❌ http://127.0.0.1:3000
  ❌ https://localhost:3000 (http여야 함)

□ 서버를 재시작했나?
  → 설정 변경 후 필수!
```

**여전히 안 될 때:**
```
1. 브라우저 캐시 삭제
2. 시크릿 모드로 테스트
3. 다른 브라우저로 시도
```

#### 5️⃣ HTTPS 미지원 에러

**증상:**
```
토스 결제 위젯:
"보안 연결(HTTPS)이 필요합니다"
```

**원인:**
```
개발 환경: http://localhost (OK)
배포 환경: http://yourdomain.com (X)

실제 결제는 HTTPS 필수!
```

**해결:**

**개발 중 (로컬):**
```
http://localhost 사용 → OK
테스트 키 사용 → OK
실제 결제 안 됨 → OK (테스트용)
```

**배포 시 (실제 서비스):**
```
1. 도메인 구매 (example.com)
2. SSL 인증서 발급 (무료: Let's Encrypt)
3. 서버에 HTTPS 설정
4. 토스 실제 키로 변경
```

**임시 해결 (개발용 HTTPS):**
```bash
# Vite 개발 서버를 HTTPS로 실행
npm install -D @vitejs/plugin-basic-ssl

# vite.config.js
import basicSsl from '@vitejs/plugin-basic-ssl'

export default {
  plugins: [basicSsl()],
  server: {
    https: true
  }
}

# 실행 → https://localhost:3000
```

### ❓ Q4: 이 방식으로 만든 사이트가 안전한가요?

**A: 기본적으로 안전하지만, 주의할 점이 있습니다**

```
✅ 안전한 부분:
- 토스 페이먼츠가 결제 처리
  → 카드 정보는 토스 서버에만 전달
  → 우리 서버는 카드 정보 저장 안 함
- API 키를 .env 파일에 저장
  → 코드에 직접 노출 안 함
- 서버에서 결제 승인
  → 클라이언트에서 조작 불가능

⚠️ 주의할 부분:
- .env 파일 Git에 업로드 금지!
  → .gitignore에 추가 필수
- 서비스 계정 키 파일 보호
  → 절대 공개하지 말 것
- 구글 시트 공유 설정
  → 서비스 계정에만 공유
  → 모든 사람에게 공유 금지
```

**보안 체크리스트:**
```
배포 전 확인:
□ .env 파일이 .gitignore에 있는지
□ service-account-key.json도 gitignore에
□ 테스트 키를 실제 키로 교체했는지
□ 구글 시트 공유 권한 확인
□ HTTPS로 접속 가능한지
□ CORS 설정이 특정 도메인만 허용하는지
```

## 용어 설명

### 🔤 A-Z 순서 용어 사전

#### A

**API (Application Programming Interface)**
- **쉬운 설명**: 프로그램 간 대화 규칙
- **비유**: 식당 메뉴판
  - 손님: "불고기 정식 주세요" (API 요청)
  - 주방: [음식 준비] (API 처리)
  - 손님: [음식 받음] (API 응답)
- **예시**: 토스 결제 API
  ```
  요청: "이 카드로 12,000원 결제해주세요"
  응답: "결제 완료되었습니다. 거래번호는 ABC123"
  ```

**API 키 (API Key)**
- **쉬운 설명**: 서비스 사용 허가증
- **비유**: 아파트 출입증
- **예시**: test_ck_abc123xyz
- **주의**: 비밀번호처럼 안전하게 보관!

#### B

**백엔드 (Backend)**
- **쉬운 설명**: 눈에 안 보이는 서버 쪽 작업
- **비유**: 식당의 주방
- **담당 업무**:
  - 데이터베이스에서 정보 가져오기
  - 결제 처리
  - 비밀 정보 관리
- **기술**: Node.js, Express

#### C

**클라이언트 (Client)**
- **쉬운 설명**: 사용자가 보는 화면
- **비유**: 식당 홀 (손님이 있는 공간)
- **담당 업무**:
  - 예쁜 디자인 보여주기
  - 버튼 클릭 처리
  - 사용자 입력 받기
- **기술**: React, HTML, CSS

**CORS (Cross-Origin Resource Sharing)**
- **쉬운 설명**: 다른 주소끼리 통신 허용 규칙
- **비유**: 출입국 심사
- **문제 상황**:
  ```
  클라이언트(localhost:3000)가
  서버(localhost:4000)에 요청
  → 서버가 거부!
  ```
- **해결**: 서버에서 허용 설정
  ```javascript
  app.use(cors({
    origin: 'http://localhost:3000'
  }))
  ```

**Cursor (커서)**
- **쉬운 설명**: AI가 탑재된 코드 편집기
- **특징**: ChatGPT처럼 대화하며 코드 작성
- **사용법**: @ 기능으로 파일/문서 참조

#### D

**디버깅 (Debugging)**
- **쉬운 설명**: 버그(에러) 찾아 고치기
- **비유**: 고장난 자동차 수리
- **과정**:
  1. 어디가 문제인지 찾기
  2. 왜 문제인지 파악
  3. 수정
  4. 다시 테스트

**데이터베이스 (Database, DB)**
- **쉬운 설명**: 데이터를 저장하는 창고
- **비유**: 도서관 (책들이 정리된 곳)
- **이 실습에서**: 구글 시트 사용
- **실제 서비스**: MySQL, PostgreSQL 등

#### E

**env (Environment Variables, 환경 변수)**
- **쉬운 설명**: 비밀 설정값 저장 파일
- **파일명**: .env (점으로 시작!)
- **내용 예시**:
  ```
  API_KEY=test_sk_abc123
  PORT=4000
  ```
- **중요**: Git에 업로드 금지!

**Express**
- **쉬운 설명**: Node.js 웹 서버 프레임워크
- **역할**: 서버 쉽게 만들기
- **비유**: 집 짓기 세트
  - 날것: 나무, 못, 시멘트 (Node.js)
  - 세트: 조립식 가구 (Express)

#### F

**프론트엔드 (Frontend)**
- **쉬운 설명**: 사용자가 보고 만지는 부분
- **클라이언트**와 같은 의미
- **기술**: React, Vue, HTML/CSS/JS

#### G

**Git**
- **쉬운 설명**: 코드 버전 관리 도구
- **비유**: 게임 저장
  - 버전 1: 기본 기능
  - 버전 2: 결제 추가
  - 버전 3: 구글 시트 연동
- **명령어**:
  ```bash
  git add .        # 변경사항 스테이징
  git commit -m    # 저장
  git push         # 업로드
  ```

#### H

**HTTP / HTTPS**
- **HTTP**: 웹 통신 규약
- **HTTPS**: 보안 웹 통신 (S=Secure)
- **비유**:
  - HTTP: 엽서 (내용 다 보임)
  - HTTPS: 봉인된 편지 (암호화됨)
- **중요**: 실제 결제는 HTTPS 필수!

#### J

**JSON**
- **쉬운 설명**: 데이터 정리 형식
- **비유**: 엑셀 표를 텍스트로
- **예시**:
  ```json
  {
    "이름": "김철수",
    "나이": 25,
    "취미": ["독서", "음악"]
  }
  ```

#### L

**localhost**
- **쉬운 설명**: 내 컴퓨터 주소
- **주소**: http://localhost:3000
- **의미**: "서버가 내 컴퓨터에 있어요"
- **배포 후**: example.com (실제 도메인)

#### N

**Node.js**
- **쉬운 설명**: JavaScript로 서버 만드는 도구
- **원래**: JS는 브라우저에서만 작동
- **Node.js 등장**: JS로 서버도 만들 수 있음!
- **설치 확인**: `node --version`

**npm (Node Package Manager)**
- **쉬운 설명**: Node.js 프로그램 설치 도구
- **비유**: 앱 스토어
- **명령어**:
  ```bash
  npm install express  # Express 설치
  npm start           # 프로그램 실행
  npm run dev         # 개발 모드 실행
  ```

#### P

**포트 (Port)**
- **쉬운 설명**: 컴퓨터의 문 번호
- **비유**: 아파트 호수
  - 301호: 웹 서버
  - 302호: 데이터베이스
  - 303호: API 서버
- **이 실습**:
  - 포트 3000: 클라이언트
  - 포트 4000: 서버

**프롬프트 (Prompt)**
- **쉬운 설명**: AI에게 주는 명령/질문
- **중요도**: ⭐⭐⭐⭐⭐
- **좋은 프롬프트**: 상세하고 구체적
- **나쁜 프롬프트**: "결제 만들어줘"
- **좋은 프롬프트**: "@문서 @샘플 참고해서 CORS 설정하고 포트 3000으로 결제 구현해줘"

#### R

**React**
- **쉬운 설명**: 웹 UI 만드는 라이브러리
- **특징**: 컴포넌트 기반
- **비유**: 레고 블록
  - 버튼 블록
  - 입력창 블록
  - 이미지 블록
  → 조합해서 화면 완성

**Rubber (러버)**
- **쉬운 설명**: 노코드 웹 디자인 도구
- **특징**: 드래그 앤 드롭으로 UI 제작
- **이 실습**: 1주차에 사용

#### S

**서버 (Server)**
- **쉬운 설명**: 데이터를 관리하는 컴퓨터
- **역할**:
  - 데이터 저장/조회
  - 비밀 정보 보관
  - 복잡한 계산 처리
- **이 실습**: Express 서버

**SDK (Software Development Kit)**
- **쉬운 설명**: 개발 도구 세트
- **비유**: 조립 가구 키트
- **예시**: 토스 페이먼츠 결제 위젯 SDK
- **포함**: 코드, 문서, 샘플

**Syntax Error (신택스 에러)**
- **쉬운 설명**: 문법 오류
- **예시**:
  ```javascript
  const name = "홍길동  // 따옴표 안 닫음!
  function test() {    // 중괄호 안 닫음!
  ```
- **특징**: 스스로 수정 가능

#### T

**터미널 (Terminal)**
- **쉬운 설명**: 명령어로 컴퓨터 조작
- **Mac**: Terminal 앱
- **Windows**: 명령 프롬프트, PowerShell
- **Cursor**: 하단에 내장됨

**토스 페이먼츠 (Toss Payments)**
- **쉬운 설명**: 한국 온라인 결제 서비스
- **제공**: 결제 위젯, API
- **지원**: 카드, 계좌이체, 간편결제

#### V

**Vite (비트)**
- **쉬운 설명**: 빠른 웹 개발 도구
- **역할**: React 앱을 실행하고 빌드
- **명령어**: `npm run dev`

#### W

**위젯 (Widget)**
- **쉬운 설명**: 작은 UI 컴포넌트
- **예시**: 결제 위젯
  ```
  ┌─────────────────┐
  │ 결제 수단 선택   │
  │ ⚪ 카드         │
  │ ⚪ 계좌이체      │
  │ ⚪ 간편결제      │
  └─────────────────┘
  ```

#### 기타

**@ 기능 (Cursor)**
- **쉬운 설명**: 파일/URL 참조하기
- **사용법**: `@파일명` 또는 `@URL`
- **효과**: AI가 해당 내용을 직접 읽음
- **예시**: `@https://docs.tosspayments.com`
