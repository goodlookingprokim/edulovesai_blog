🔄 다양한 구분자 사용하기 {#다양한-구분자-사용하기}

구분자란? {#구분자란}

배열의 값들을 구분하는 기호입니다. 기본값은 쉼표(,) 이지만, 다른 것도 사용할 수 있습니다.

구분자의 3가지 종류 {#구분자의-3가지-종류}

1. 쉼표 (,) - 기본값

Code 
친구[3]: 철수,영희,민준
1 line

장점: 가장 흔하고 읽기 쉬움 단점: 데이터에 쉼표가 있으면 따옴표 필요

2. 탭 (Tab) - \t

Code 
친구[3	]:	철수	영희	민준
1 line

장점: 토큰을 더 절감할 수 있음 단점: 화면에 보기 어려울 수 있음

3. 파이프 (|) - 명확함

Code 
친구[3|]: 철수|영희|민준
1 line

장점: 명확하고 쉼표 문제 없음 단점: 파이프 자체가 데이터에 있으면 따옴표 필요

구분자 선택 가이드 {#구분자-선택-가이드}

상황 추천 구분자 이유
쉼표가 없는 단순 데이터 , 가장 일반적
데이터에 쉼표가 있음 | 따옴표 피할 수 있음
토큰을 최대한 절감하고 싶을 때 \t 가장 효율적
데이터 검증이 중요할 때 | 명확함

예제 1: 주소 데이터 (쉼표 포함) {#예제-1-주소-데이터-쉼표-포함}

❌ 쉼표 사용 (비효율적):

Code 
주소[2]: "서울, 강남구","부산, 해운대구"
1 line

✅ 파이프 사용 (효율적):

Code 
주소[2|]: 서울, 강남구|부산, 해운대구
1 line

예제 2: 성능 비교 - 학생 100명 정보 {#예제-2-성능-비교-학생-100명-정보}

쉼표 구분자:

Code

  이영희,19,2,2
  박민준,21,3,3
  ...
5 lines

탭 구분자:

Code 
학생[100	]{이름	나이	학년	반}:

    박민준	21	3	3

**토큰: 약 420개 (16% 절감!)**

🗂️ Key Folding으로 토큰 절감 {#key-folding으로-토큰-절감}

Key Folding이란? {#key-folding이란}

중첩된 객체의 경로를 점(.)으로 연결하는 기능입니다.

목적: 들여쓰기를 줄여서 토큰을 절감

비교: 일반 vs Key Folding {#비교-일반-vs-key-folding}

일반 형태 (들여쓰기 사용)

Code 
사용자:
  프로필:
    이름: 김철수
    나이: 30
    주소:
      시: 서울
      구: 강남구
7 lines

토큰: 약 60개

Key Folding 형태 (점 사용)

Code 
사용자.프로필.이름: 김철수
사용자.프로필.나이: 30
사용자.프로필.주소.시: 서울
사용자.프로필.주소.구: 강남구
4 lines

토큰: 약 50개 (17% 절감!)

Key Folding 사용 규칙 {#key-folding-사용-규칙}

규칙 1: 단일 자식만 접으면 됨

✅ 접을 수 있음:

Plain Text 
데이터 → 학생 → 이름
↓
데이터.학생.이름
3 lines

❌ 접으면 안 됨 (형제가 여럿):

Plain Text 
학생 (이름, 나이 두 개가 형제)
    - 이름
    - 나이
3 lines

규칙 2: 자동 복원 가능

Key Folding으로 접은 데이터는 다시 원래 형태로 복원할 수 있습니다.

Bash 
# Key Folding으로 저장
npx @toon-format/cli input.json --key-folding safe -o output.toon


다시 원래 형태로 복원


npx @toon-format/cli output.toon --expand-paths safe -o restored.json
4 lines

객체, 배열, 표가 섞여 있는 복잡한 데이터 구조입니다.

    1,3,90
    2,3,85
    3,3,95

    시도: 서울

11 lines

Plain Text 
학교 (객체)

├── 설립일: 단순 값
├── 학년 (표)
│   ├── 학년 | 반수 | 학생수
│   ├── 1 | 3 | 90
│   ├── 2 | 3 | 85
│   └── 3 | 3 | 95

    └── 상세: 테헤란로 123

중요한 고려사항 {#중요한-고려사항}

"TOON은 토큰을 절감한다" = "항상 더 빠르다" 는 잘못된 가정입니다!

시나리오: JSON vs TOON 비교 {#시나리오-json-vs-toon-비교}

클라우드 API (OpenAI, Claude) 사용

Plain Text 
데이터 크기: 5,000 토큰
JSON:  5,000 토큰 → 2초
TOON:  2,000 토큰 → 1.5초 ✅ 더 빠름!


이유: 네트워크 전송 시간이 큼
4 lines

TOON이 이점: 토큰이 적으면 전송 시간이 짧음

로컬 모델 (Ollama, LLaMA) 사용

Plain Text 
데이터 크기: 5,000 토큰
JSON:  5,000 토큰 → 2초
TOON:  2,000 토큰 → 2.5초 ❌ 더 느림!


이유: 파싱 오버헤드
4 lines

주의: TOON 파싱이 추가 시간 소요

언제 어떤 것을 써야 하나? {#언제-어떤-것을-써야-하나}

✅ TOON을 추천

Plain Text 
상황: OpenAI / Claude / Anthropic 사용
- 네트워크 전송이 병목
- 비용이 중요
- 응답 시간이 중요
4 lines

✅ JSON을 추천

  ...
4 lines

Q: 모든 데이터를 TOON으로 변환해야 하나요? {#q-모든-데이터를-toon으로-변환해야-하나요}

A: 아니요. 테이블 형식의 균일한 데이터에는 TOON이 좋지만, 깊게 중첩된 데이터는 JSON이 더 나을 수 있습니다.

Q: 기존 JSON을 TOON으로 자동 변환할 수 있나요? {#q-기존-json을-toon으로-자동-변환할-수-있나요}

A: 네! CLI 도구를 사용하면 자동 변환됩니다: npx @toon-format/cli input.json -o output.toon

Q: TOON으로 변환한 데이터를 다시 JSON으로 돌릴 수 있나요? {#q-toon으로-변환한-데이터를-다시-json으로-돌릴-수-있나요}

A: 네! 양방향 변환이 가능합니다: npx @toon-format/cli input.toon -o output.json

Q: 매우 큰 데이터셋의 경우는? {#q-매우-큰-데이터셋의-경우는}
